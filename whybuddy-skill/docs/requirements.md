# WhyBuddy Skill 需求说明

## 目标

把"用户想法 / 仓库 / 文件 / 截图"收敛成一套可规划、可校验、可预览、可交付、可回炉的 WhyBuddy 闭环 Skill。

## 范围

- 输入归一化与证据抽取
- 缺失信息澄清与就绪判定
- 多路线生成、比较、选择与确认
- 单 Agent / 多角色协作模式决策
- 规格树生成、校验、兜底与来源追踪
- 文档派生、效果预览、交付包导出
- 评审反馈、重规划、失效传播与自动重算
- 质量门与转人工出口

## 非范围

- 事件总线、Socket、状态仓、前端画布、工作台 UI
- GitHub 权限托管与账号体系
- 实际的模型服务、队列系统、容器基础设施

## 输入要求

- 必须接受一句话目标作为最小输入
- 可接受仓库 URL、代码文件、截图、日志、补充限制
- 当仓库不可访问时，必须保留失败状态并继续流程
- 输入归一化后必须保留证据来源

## 功能要求

### FR-1 输入层

- 必须检测是否存在仓库链接
- 有仓库链接时才执行仓库解析
- 归一化结果必须包含目标、摘要、来源、证据
- 权限失败必须走降级支路，不得让流程中断

### FR-2 澄清层

- 必须区分阻塞型与非阻塞型缺失信息
- 必须支持生成澄清问题并回收答案
- 就绪闸必须具备"通过"与"未就绪退回"两条支路
- 澄清结果必须沉淀为成功标准

### FR-3 路线规划

- 至少生成三类路线：标准、深度、升级
- 必须比较风险、成本、收益与适用条件
- 路线选择后必须经过轻量确认闸
- 确认闸被拒绝时必须回退到路线选择

### FR-4 决策与协作

- 必须在路线确认后判断任务复杂度
- 简单任务必须支持单 Agent 直出
- 复杂任务必须支持多角色头脑风暴
- 单 Agent 与多角色都必须能访问工具代理
- 工具故障、编排异常、超时必须降级回单 Agent

### FR-5 规格树生成

- 必须独立执行脱敏步骤
- LLM 输出必须先过 Schema 校验
- 归一化后必须再过不变量守卫
- 不变量至少覆盖：唯一根、父可达、深度、无环、类型合法
- LLM 必须支持一次重试
- 重试失败必须进入确定性兜底
- 成功与兜底都必须写入来源追踪

### FR-6 文档派生

- 必须生成 `requirements.md`、`design.md`、`tasks.md`
- 验收标准必须能回链到澄清阶段的成功标准
- 文档生成结果必须能被预览与交付阶段复用

### FR-7 预览与交付

- 必须生成提示词包
- 必须生成效果预览描述
- 必须输出交付包导出描述
- 预览不满意必须能进入反馈闭环

### FR-8 评审与反馈

- 评审闸必须定义"通过"与"回炉"
- 反馈必须支持进入重规划
- 重规划必须支持回退到澄清、路线、规格树、模式决策
- 超预算或不收敛必须支持转人工

### FR-9 失效与依赖

- 必须维护上游变更到下游影响的依赖关系
- 必须记录失效索引，至少包含 `staleSince`、`reason`、`fromStage`
- 必须支持自动重算规格树、文档与预览
- 重规划触发的上游修改必须同步导致下游失效

### FR-10 状态与质量门

- 节点状态必须由单一派生器生成
- 实时状态与回放必须按 `sessionId` 隔离
- 必须同时支持测试质量门与内容质量门
- 合并门必须由自动断言与人工目检共同决定

## 非功能要求

- 流程闭环完整，所有判断闸都有拦下支路
- 关键确定性检查可由脚本稳定复现
- 失败状态可观察、可回放、可解释
- 文档命名稳定，便于宿主消费
- 产物具备来源追踪，便于审计

## 验收标准

- 给定最小输入时，Skill 能生成项目上下文、澄清简报、路线结果、规格树、三份文档和交付描述
- 任一判断闸都存在明确的 no-path 处理
- LLM 输出非法时，Skill 能在一次重试后兜底生成合法规格树
- 上游变更后，Skill 能标记下游失效并重建受影响产物
- 评审拒绝后，Skill 能按问题层级重规划，而不是只退一步

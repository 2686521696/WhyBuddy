# 参数采集节点现状核查

## 结论

截至 2026-04-22，`web-aigc-node-param_collection` 在当前主仓中的实际状态仍然是：

- 可以勾选 1 项
- 暂不能勾选 3 项

也就是说，`tasks.md` 的主判断方向是准确的，这次不需要调整勾选结果，只需要把证据表达得更贴近当前代码事实。

当前最准确的描述不是“参数采集节点已经完整落地”，而是：

- 已复用通用 HITL waiting / resume 底座
- 已具备字段结构、`formData`、`inputSchema` 的共享契约承接位
- 已具备通用 waiting / decision / runtime / lineage 事件链路
- 但仍未形成 `param_collection` 专属的多字段表单、字段校验、附件采集与专用事件闭环

## 任务项核查

### 1. 定义字段结构与校验规则

暂不能勾选。

已具备的事实：

- `shared/mission/contracts.ts`
  - `WEB_AIGC_HITL_NODE_TYPES` 已包含 `param_collection`
  - `WebAigcHitlFieldDefinition` 已定义：
    - `key`
    - `label`
    - `type`
    - `required`
    - `placeholder`
    - `defaultValue`
    - `options`
  - `WebAigcHitlSubmissionMetadata.formData` 已提供参数回写位置
- `shared/workflow-runtime-engine.ts`
  - `WorkflowNodeWaitResult.inputSchema` 已支持等待节点回传字段 schema
- `shared/workflow-domain.ts`
  - `WebAigcFieldSchema`
  - `WebAigcGraphCheckpoint.inputSchema`
  已支持图运行时保存等待输入字段描述

仍然缺少的事实：

- 没有看到 `param_collection` 节点专用字段校验器
- `server/tasks/mission-decision.ts` 当前仍主要校验：
  - `optionId`
  - `freeText`
  - `requiresComment`
- 没有看到基于 `WebAigcHitlFieldDefinition` 或 `WebAigcFieldSchema` 的服务端多字段校验
- 没有看到字段级错误、默认值回填、数字/布尔/枚举校验的专项测试

结论：

字段结构契约已经存在，但“字段校验规则落地”还不够，不能勾选。

### 2. 打通等待输入与恢复执行

可以勾选。

已具备的事实：

- `server/tasks/mission-store.ts`
  - `markWaiting()` 会把任务置为 `waiting`
  - `resolveWaiting()` 会把任务恢复为 `running`
  - `decisionHistory` 会持久化
- `server/tasks/mission-runtime.ts`
  - 提供 `waitOnMission()`
  - 提供 `resumeMissionFromDecision()`
  - 提供 `decisionSubmitted` socket 广播
- `server/tasks/mission-decision.ts`
  - 处理等待态决策提交
  - 处理超时校验
  - 提交后恢复执行
- `server/routes/tasks.ts`
  - 已提供 `POST /api/tasks/:id/decision`
- `client/src/components/tasks/DecisionPanel.tsx`
  - 已接入等待态决策提交面板
- `client/src/lib/mission-client.ts`
  - 已接入决策提交 API
- `client/src/lib/tasks-store.ts`
  - 已接入 decisionSubmitted socket 结果消费
- `server/core/workflow-runtime-engine.ts`
  - wait 结果会进入 `instance.checkpoint`
  - `result.inputSchema` 会写入 `checkpoint.inputSchema`
  - wait 时会发出 `node.waiting_input`

已有验证：

- `server/tests/hitl-decision.test.ts`
- `server/tests/mission-routes.test.ts`
- `server/tests/workflow-runtime-engine.test.ts`

结论：

无论从 mission 路由链路，还是从 graph runtime checkpoint 角度看，等待输入与恢复执行都已经形成闭环，这一项可以继续保持已完成。

### 3. 写入参数采集事件

暂不能勾选。

已具备的事实：

- `shared/web-aigc-observability.ts`
  - 已定义 `node.waiting_input`
  - 已定义 `human.decision_submitted`
- `server/core/workflow-runtime-engine.ts`
  - wait 节点会写 `checkpoint.inputSchema`
  - wait 节点会发出 `node.waiting_input`
- `server/tasks/mission-runtime.ts`
  - 决策提交后会广播 `decisionSubmitted`
- `server/audit/audit-hooks.ts`
  - 已把决策提交通用映射为 `human.decision_submitted` 审计事件
- `server/tasks/mission-decision.ts`
  - 成功提交后会把以下 metadata 写入 decision lineage：
    - `nodeType`
    - `sessionId`
    - `interactionId`
    - `branchKey`
    - `formData`

仍然缺少的事实：

- 没有看到 `param_collection` 专用事件模型
- 没有看到专门区分“普通 HITL 决策”和“结构化参数采集”的事件语义
- 没有看到对 `formData`、字段列表、字段摘要的专门审计事件
- 没有看到 `nodeType = "param_collection"` 的专项事件测试

结论：

现在能证明的是“通用 waiting / decision / runtime / lineage 事件链路存在”，但还不能证明“参数采集事件已经独立落账完成”，因此这一项仍不应勾选。

### 4. 验证多字段与附件场景

暂不能勾选。

已具备的事实：

- `shared/mission/contracts.ts`
  - `formData` 已允许回传结构化键值
- 仓库其他模块存在附件处理基础
  - 这说明平台总体上不是完全没有附件能力

仍然缺少的事实：

- `client/src/components/tasks/DecisionPanel.tsx` 当前仍然是：
  - 选项按钮
  - 注释文本
  - 自由文本输入
- 没有看到多字段动态表单渲染
- 没有看到附件上传 UI
- `WebAigcHitlSubmissionMetadata.formData` 当前只支持：
  - `string`
  - `number`
  - `boolean`
  - `null`
  不支持附件对象
- 没有看到 `param_collection` 的多字段提交测试
- 没有看到附件型参数采集测试

结论：

这项仍明显未完成，保持未勾选是准确的。

## 对 `tasks.md` 的最终判断

本次核查后，`tasks.md` 的勾选结果不需要改动，仍然应保持：

- `定义字段结构与校验规则`：未完成
- `打通等待输入与恢复执行`：已完成
- `写入参数采集事件`：未完成
- `验证多字段与附件场景`：未完成

本次仅建议补充说明两点，且已在 `tasks.md` 中同步：

1. `inputSchema` 已经真实进入 runtime checkpoint，而不只是共享类型存在
2. `formData` 目前已进入 decision lineage metadata，但这还不能等同于参数采集专用事件闭环

## 当前最小阻塞点

如果后续要把这个 spec 再推进，最小缺口主要有 4 个：

1. 为 `param_collection` 增加真正可执行的多字段 schema 校验器
2. 在 `DecisionPanel` 或新的表单组件中渲染多字段动态表单
3. 为 `formData` 增加参数采集专用事件与审计摘要
4. 补多字段与附件场景的专项测试

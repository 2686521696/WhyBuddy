# 参数采集节点落地现状

## 一、总体判断

`web-aigc-node-param_collection` 当前并没有一套独立的“参数采集节点实现”，而是主要复用现有的 HITL / mission decision / waiting-resume 能力。

因此，仓库现状更接近：

- 已有“等待用户输入并恢复执行”的通用底座
- 已有一部分“字段结构契约”的预留
- 但尚未形成完整的“结构化多字段参数采集节点”闭环

## 二、已落地能力

### 1. 等待输入与恢复执行已经打通

这部分证据最完整：

- `server/tasks/mission-store.ts`
  - `markWaiting()` 会把任务置为 `waiting`
  - `resolveWaiting()` 会把任务恢复为 `running`
  - `decisionHistory` 会保留下来
- `server/tasks/mission-runtime.ts`
  - `waitOnMission()`
  - `resumeMissionFromDecision()`
- `server/tasks/mission-decision.ts`
  - 处理等待态决策提交
  - 校验超时、选项合法性、备注要求
  - 触发恢复执行
- `server/routes/tasks.ts`
  - 提供 `POST /api/tasks/:id/decision`
- `client/src/components/tasks/DecisionPanel.tsx`
  - 已有前端决策提交面板
- `client/src/lib/mission-client.ts`
  - 已封装决策提交 API

相关测试也比较完整：

- `server/tests/hitl-decision.test.ts`
- `server/tests/mission-routes.test.ts`
- `server/tests/workflow-runtime-engine.test.ts`

### 2. 字段结构契约已有预留

虽然还没有完整落地，但仓库里已经有一些关键契约：

- `shared/mission/contracts.ts`
  - `WebAigcHitlFieldDefinition`
  - `WebAigcHitlSubmissionMetadata.formData`
  - `WEB_AIGC_HITL_NODE_TYPES` 中已经包含 `param_collection`
- `shared/workflow-runtime-engine.ts`
  - `WorkflowNodeWaitResult.inputSchema`
- `shared/workflow-domain.ts`
  - `WebAigcFieldSchema`
  - `WebAigcGraphCheckpoint.inputSchema`

这说明平台已经为“节点进入等待态时携带字段定义”留好了类型位置。

### 3. 通用事件链路已有底座

仓库已经存在等待/决策类事件链路：

- `server/tasks/mission-store.ts`
  - 写入 `waiting` / `progress` 事件
- `shared/web-aigc-observability.ts`
  - 定义了 `node.waiting_input`
  - 定义了 `human.decision_submitted`
- `server/audit/audit-hooks.ts`
  - 将人类决策提交映射为审计事件

这意味着参数采集节点未来可以复用现有的 observability / audit 基座。

## 三、未完成部分

### 1. 结构化字段校验没有真正落地

当前虽然有字段定义类型，但没有看到：

- 多字段表单的服务端校验器
- 字段必填错误返回
- 数字/布尔/枚举校验
- 默认值注入
- 基于字段 schema 的前端动态渲染

现有 `server/tasks/mission-decision.ts` 只处理：

- `optionId`
- `freeText`
- `requiresComment`

它还不是 `param_collection` 所需的“多字段参数校验器”。

### 2. 参数采集事件还不是专用模型

现有事件更多是通用 HITL 事件，而不是参数采集专用事件。

当前缺少：

- 参数字段列表摘要
- `formData` 提交摘要
- 字段级采集完成事件
- `nodeType = param_collection` 的专项事件验证

所以目前不能说“参数采集事件已经写入完成”，只能说“通用等待/决策事件底座已存在”。

### 3. 多字段与附件场景没有形成闭环

这一块缺口最明显：

- `client/src/components/tasks/DecisionPanel.tsx` 目前只有选项与文本输入
- 没有附件上传入口
- `formData` 只允许基础标量，不支持附件对象
- 没有 `param_collection` 多字段测试
- 没有 `param_collection` 附件测试

附件能力目前主要存在于 workflow 输入链路，而不是 mission decision / param collection 链路。

## 四、任务项判断

### 1. 定义字段结构与校验规则

- 不建议勾选
- 原因：字段结构契约存在，但字段校验实现和验证不足

### 2. 打通等待输入与恢复执行

- 可以勾选
- 原因：mission waiting / decision / resume 与 route / client / runtime 已形成闭环

### 3. 写入参数采集事件

- 不建议勾选
- 原因：只有通用等待/决策事件，没有参数采集专用事件模型和专项验证

### 4. 验证多字段与附件场景

- 不建议勾选
- 原因：没有多字段动态表单与附件型参数采集的实现闭环和测试

## 五、后续最小补齐建议

如果要把这个 spec 真正推进到可落地状态，建议按下面顺序补：

1. 为 `param_collection` 定义真正可执行的字段 schema
   - 包括字段类型、必填、默认值、校验规则、枚举选项
2. 在 mission decision 或 graph runtime adapter 层增加字段校验
   - 不能只靠 `freeText` 与 `optionId`
3. 把 `formData` 变成可审计、可回放的标准载荷
   - 至少记录字段 key、脱敏摘要、节点类型
4. 补多字段和附件测试
   - 多字段必填
   - 数值/布尔/枚举字段
   - 附件型参数采集
   - 提交后恢复执行

## 六、结论

`web-aigc-node-param_collection` 当前最准确的描述是：

- 复用了现有 HITL waiting / resume 底座
- 具备字段结构的类型预留
- 尚未具备完整的结构化参数采集节点实现

所以这条 spec 目前只能认定“等待恢复闭环已完成”，其余任务仍需继续落地。

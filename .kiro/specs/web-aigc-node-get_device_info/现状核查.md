# 现状核查：获取设备信息节点

## 结论

`get_device_info` 已在主仓形成低风险最小闭环，但能力边界被刻意收敛为“摘要级宿主环境信息”。

## 已落地能力

- 输出结构已统一为 `status / runtime / client / privacy / compatibility / context / warnings`
- 运行时摘要来自 Node 宿主环境，仅暴露 `platform / arch / nodeVersion`
- 客户端摘要仅接受 `userAgent / platform / locale / timezone / appVersion / screenCategory`
- `userAgent` 只用于解析浏览器族与操作系统族，不原样落库
- 隐私策略固定为 `summary_only`，显式标记 `rawUserAgentStored = false`，并支持 `allowRuntimeDetails / allowClientHints / redactUserAgent / retention` 做摘要级控制
- 路由支持从请求头补齐 `User-Agent / X-Client-Platform / X-Client-Locale / X-Client-Timezone`
- 当客户端 hints 缺失或被策略关闭时，会降级为仅返回运行时摘要，并给出 warning 提示

## 当前边界

- 不采集 IP、hostname、设备序列号、MAC、精确硬件型号
- 不尝试浏览器指纹、屏幕分辨率、插件清单、持久化设备标识
- 兼容性验证口径是“服务端可消费宿主摘要 + 前端可通过请求头透传提示”，不是完整前端 SDK

## 建议

保持当前收敛版实现，后续只有在 Office 端确实出现设备差异化编排场景时，再扩展更细的客户端 hints。

# Web-AIGC 平台可观测与审计现状核查

## 核查结论

截至 2026-04-22，`.kiro/specs/web-aigc-platform-observability-audit/tasks.md` 当前工作区中的 5 个任务项仍有对应实现或测试证据支撑，现阶段可以保留为已完成状态；但需要把“索引模型已定义”和“runtime 事件已实际打通的落账范围”明确区分开来。

本次核查没有再改写 `tasks.md`，原因是该文件已经存在并行修改，且当前勾选状态与代码现状不冲突。为避免覆盖他人工作，本次仅补充中文核查文档，作为任务勾选的依据说明。

## 任务项逐条核对

### 1. 定义节点级事件类型与载荷

结论：可勾。

依据：

- `shared/web-aigc-observability.ts` 已集中定义 Web-AIGC 事件目录，覆盖 `node.started`、`node.completed`、`node.failed`、`node.waiting_input`。
- 上述事件均声明了 `requiredFields`，包含 `workflowId`、`instanceId`、`nodeId`、开始/结束时间、状态、错误或等待信息等关键载荷。
- 这些定义直接对应 spec 中“节点级遥测”的要求。

关键文件：

- `shared/web-aigc-observability.ts`

### 2. 定义边跳转与循环事件

结论：可勾。

依据：

- `shared/web-aigc-observability.ts` 已定义 `edge.transitioned` 与 `edge.loop_iterated`。
- `requiredFields` 中包含 `edgeId`、`fromNodeId`、`toNodeId`、`kind`、`loopKey`、`iterationIndex` 等路径回放所需字段。
- `server/tests/workflow-graph-projection.test.ts` 验证图实例快照中已经可以投影出 `edgeTransitions`、节点运行态和 `waitingFor` 等回放信息。

关键文件：

- `shared/web-aigc-observability.ts`
- `server/tests/workflow-graph-projection.test.ts`

### 3. 定义人工决策与终止审计事件

结论：可勾。

依据：

- `shared/web-aigc-observability.ts` 已定义 `human.decision_submitted`、`human.approved`、`human.rejected`、`instance.terminated`、`instance.retry_requested`、`instance.escalated`、`external.vector_insert`。
- 人工决策类事件要求字段中包含 `missionId`、`decisionId`、`submittedBy` / `approvedBy` / `rejectedBy`，终止事件包含 `requestedBy` 与 `reason`。
- 本轮新增的 runtime 控制面事件也已具备明确的 `requestedBy`、`nodeId`、`waitingFor` 或 `timestamp` 等字段要求。
- `server/tests/vector-insert-adapter.test.ts` 已验证高风险向量写场景下存在 denied / approval_required 等治理分支，说明关键外部动作已进入审计关注范围。

关键文件：

- `shared/web-aigc-observability.ts`
- `server/tests/vector-insert-adapter.test.ts`

### 4. 打通 replay / audit / lineage 的关联索引

结论：可勾。

依据：

- `shared/web-aigc-observability.ts` 定义了 `workflowId`、`missionId`、`instanceId`、`sessionId`、`replayId`、`auditEntryId`、`lineageId`、`artifactId`、`nodeId`、`edgeId`、`decisionId` 等关联索引。
- `server/routes/audit.ts` 已暴露 `webAigcRelationIndexes` 和 `webAigcRelatedEntries` 接口，允许按这些索引检索关联审计项。
- `server/audit/audit-query.ts` 已实现多索引候选值读取与联合匹配逻辑，支持从 `metadata`、`metadata.links`、`context.sessionId`、`resource.id` 等位置抽取索引。
- `server/tests/audit-query.test.ts` 已验证以下场景：
  - 按 `workflowId` 命中；
  - `missionId + sessionId` 联合命中；
  - `lineageId + decisionId` 联合命中；
  - 多索引必须在同一条记录上同时成立，避免误关联。

这里要特别收紧口径：

- 当前“可勾”的依据是 replay / audit / lineage 关联索引模型与查询能力已经存在
- 这并不等于 runtime 事件已经由 `server/core/web-aigc-runtime-observability.ts` 直接写入 lineage
- 就 runtime 事件镜像现实而言，当前主仓已明确落到的是 replay / audit，两者证据最充分

关键文件：

- `shared/web-aigc-observability.ts`
- `server/routes/audit.ts`
- `server/audit/audit-query.ts`
- `server/tests/audit-query.test.ts`

### 5. 验证监控与回放界面的最小闭环

结论：可勾。

依据：

- `server/routes/replay.ts` 已提供时间轴、事件查询、导出、完整性校验、审计查询、快照创建与快照列表接口。
- 同文件中的快照创建逻辑会在写入 snapshot 后追加审计记录，说明 replay 与 audit 已形成最小闭环。
- `server/core/web-aigc-runtime-observability.ts` 已把 `web_aigc_runtime_event` 镜像到 replay / audit，包含：
  - `node.started`
  - `node.completed`
  - `node.waiting_input`
  - `node.failed`
  - `edge.transitioned`
  - `instance.terminated`
  - `instance.retry_requested`
  - `instance.escalated`
- `server/tests/web-aigc-runtime-observability.test.ts` 已验证上述 runtime 事件能落到 replay / audit。
- 上述 runtime observability 测试目前没有证明“同一批 runtime 事件已直接落到 lineage store”，因此这一层应保持谨慎表述。
- `server/tests/replay-routes.test.ts` 已验证 `POST/GET /api/replay/:missionId/snapshots` 会同时写入快照和审计条目。
- `server/tests/aigc-monitoring-routes.test.ts` 已验证监控侧能读取实例 session 详情，并优先采用 projection 中的 `sessionId` / `sourceApp`。
- `server/tests/workflow-runtime-engine.test.ts` 已覆盖等待输入、恢复执行、selection 分支、confirm_judge 分支等 HITL 回放前提。
- `server/tests/workflow-graph-projection.test.ts` 已验证任务、消息、mission 信息可投影为图实例快照，具备监控/回放界面消费的最小结构。

关键文件：

- `server/core/web-aigc-runtime-observability.ts`
- `server/tests/web-aigc-runtime-observability.test.ts`
- `server/tests/replay-routes.test.ts`
- `server/tests/aigc-monitoring-routes.test.ts`
- `server/tests/workflow-runtime-engine.test.ts`
- `server/tests/workflow-graph-projection.test.ts`

## 当前判断

当前 spec 任务项更像“平台骨架闭环已具备”，而不是“所有观测能力都已完全统一到单一运行时出口”。也就是说：

- 事件目录、索引模型、回放接口、监控读取、审计查询、关键测试都已经存在。
- 现有证据分散在 `shared`、`server/routes`、`server/audit`、`server/tests` 多个层面，但组合起来已经满足该 spec 的最小闭环目标。
- 其中 runtime 事件桥接当前可明确确认的是 `replay / audit` 两路；`lineage` 更多体现为索引模型与查询维度，而不是这条 runtime 桥接已经直写完成。

因此，`tasks.md` 当前 5 项全部勾选是有依据的。

## 仍需注意的边界

- 当前完成度更偏“最小可用闭环”，并不等于所有 Web-AIGC 节点都已经统一发出完整事件。
- 部分能力是通过投影层、查询层和测试来证明闭环，而不是通过单一总线端到端证明。
- `lineageId`、`decisionId` 等索引字段已可用于关联查询，但不应据此推断“runtime 事件已完整沉淀到 lineage 存储”。
- 若后续要把这个 spec 升级为更高置信度主线验收，建议继续补：
  - 单个真实节点运行到 replay / audit / lineage 三路同时落账的端到端测试；
  - 监控 UI 与 replay API 联动的更完整集成验证。

# Web-AIGC 平台安全与治理现状核查

## 核查结论

当前 `web-aigc-platform-security-governance` 的 `tasks.md` 已全部勾选，且与仓库里的实现和测试基本匹配。本轮核查未发现必须回调为未完成的任务项，因此未改动现有 `tasks.md` 勾选状态，只补充中文现状说明。

需要说明的是：该 spec 当前更偏“治理基线、权限矩阵、风险分级、审计兼容性”的落地，而不是要求所有高风险节点都已经完成逐一执行态接线。就任务原文而言，现有代码与测试已经足以支撑全部勾选。

- 字段中已包含：

  - `riskLevel`
  - `requiresAudit`

  - `specRefs`
- `server/routes/permissions.ts`
  已提供 `GET /api/permissions/web-aigc/matrix`

  已验证该接口能返回矩阵，且包含 `runtime.terminate`

对应需求覆盖：

- 需求 1“执行权限”中的定义管理、运行、终止、发布、恢复等分类，已通过矩阵条目表达

### 4. 定义人工审批闸门策略

结论：可保持已勾选。

依据：

- `shared/web-aigc-governance.ts`
  已在矩阵和节点风险表中使用：
  - `approvalMode: "none"`
  - `approvalMode: "embedded_hitl"`
  - `approvalMode: "manual_gate"`
- 已明确将以下高风险能力标记为人工闸门：
  - `mcp`
  - `transaction_flow`
  - `vector_insert`
  - `vector_update`
  - `vector_delete`
- `server/permission/check-engine.test.ts`
  已验证：
  - 高风险 `mcp` 调用会被治理策略阻断，并返回 `approval_required`
  - 向量写操作会命中治理策略阻断
- `server/web-aigc/vector-insert-adapter.ts`
  已在 `requireApproval` 场景返回 `approval_required`
  并输出 `governance.approval.required = true`、`status = "pending"`
- `server/tests/vector-insert-adapter.test.ts`
  已验证高风险插入场景下的审批等待结果

说明：

- 当前更能证明“审批闸门策略已经定义，并在 MCP / 向量写入等切片上被主线能力消费”
- 不代表所有高风险节点都已逐一接入同样深度的审批实现

结论：可保持已勾选。

## 当前边界与后续依赖

本 spec 维度已基本落地，但仍有几个后续依赖点：

- 并非所有高风险节点都已经完成同等深度的运行时审批 / 审计接线
- 当前最强证据集中在 `mcp`、向量写入、权限治理和审计目录兼容层
- 若后续要把治理 spec 进一步收口为“执行态全面生效”，还需要继续补：
  - `passthrough_api`
  - `message_notification`
  - `internal_api` 更细粒度高风险写操作
  - 与任务阻断、人工批准、回放跳转的专项集成测试

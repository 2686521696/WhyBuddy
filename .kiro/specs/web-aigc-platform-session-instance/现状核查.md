### 4. 打通强制终止与任务取消链路

结论：可保持已勾选。

依据：

- `server/routes/aigc-monitoring.ts`
  已提供 `POST /api/v1/:tenantId/aigc-monitoring/instances/:instanceId/terminate`
- 该路由不会自行造一套新终止逻辑，而是调用：
  `operatorService.submit(missionId, { action: "terminate", ... })`
- `server/tasks/mission-operator-service.ts`
  中的 `terminateMission(...)` 会：
  - 先调用 `forwardTerminateToExecutor(...)`
  - 再复用 `runtime.cancelMission(...)`
  - 同时记录 operator action
- `server/tests/aigc-monitoring-routes.test.ts`
  已验证监控终止接口会复用 mission terminate flow
- `server/tests/mission-operator-actions.test.ts`
  已验证 `terminate` 会复用取消执行链路

说明：

- 这里的设计是“监控侧强制终止”在兼容层返回 `FORCE_TERMINATED`
- 底层任务侧则复用已有 cancel / executor cancel 机制
- 这与 spec 中“终止能力复用任务取消与执行器取消链路”的设计目标一致

## 当前边界与后续依赖

当前已完成的是“兼容层最小闭环”，后续仍有一些可以继续增强的点：

- 当前实例详情主要来自 workflow / mission / graph snapshot 投影，不是独立的新实例存储
- `auditId` 在详情链接中仍可能为空，说明实例、审计、回放三者的深层链接还有继续增强空间
- internal API 已接入实例详情与会话详情，但更完整的 operator 监控 API 组合能力仍可继续补强
- 若后续要支撑更复杂的监控 UI，还可以继续补：
  - 更多筛选维度
  - 更细粒度节点报错 / 变量快照
  - 终止前后状态变化的专项集成测试

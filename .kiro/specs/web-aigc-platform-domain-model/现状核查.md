# Web-AIGC 平台领域模型现状核查

## 核查结论

截至 2026-04-22，`web-aigc-platform-domain-model/tasks.md` 当前 5 个任务项都仍有明确代码或文档产物支撑，可以继续保持已勾选状态，本轮无需下调勾选。

本次核查仅对照当前共享契约、测试与模板产物做事实校准，不改动任何 `server/client` 代码，也不触碰其它 spec。

## 逐项核对

### 1. 定义编排定义、版本、实例、节点运行的共享类型

结论：可继续保持已完成。

依据如下：

- 共享领域模型已经在 `shared/workflow-domain.ts` 中定义：
  - `WebAigcGraphDefinition`
  - `WebAigcGraphVersion`
  - `WebAigcGraphInstance`
  - `WebAigcNodeSchema`
  - `WebAigcEdgeSchema`
  - `WebAigcNodeRunRecord`
  - `WebAigcEdgeTransitionRecord`
  - `WebAigcGraphCheckpoint`
  - `StoredWebAigcRuntimeState`
- 运行时适配器契约 `shared/workflow-runtime-engine.ts` 已直接复用这些共享类型，说明它们不是“只存在于设计稿里”，而是已经成为真实代码契约。

这意味着平台级领域模型已经形成统一命名和统一结构，不存在“每个模块各自造一套 graph instance / node run 结构”的情况。

### 2. 定义状态映射表与状态转换规则

结论：可继续保持已完成。

依据如下：

- `shared/workflow-domain.ts` 已统一定义运行态：
  - `PENDING`
  - `EXECUTING`
  - `WAITING_INPUT`
  - `EXECUTED`
  - `EXCEPTION`
  - `FORCE_TERMINATED`
- 同文件已实现状态映射函数：
  - `toCubeWorkflowStatus()`
  - `toWebAigcRuntimeStatus()`
  - `toWebAigcNodeRunStatus()`
- `shared/__tests__/workflow-domain.test.ts` 已覆盖：
  - Cube workflow 状态到统一运行态的映射
  - task 状态到 node run 状态的映射
  - 统一运行态回写到 Cube workflow 状态的映射

因此，这一项不仅有类型枚举，还有明确的映射函数和测试支撑。

### 3. 定义节点输入输出和配置元模型

结论：可继续保持已完成。

依据如下：

- `shared/workflow-domain.ts` 中的 `WebAigcFieldSchema` 已统一约束字段元信息：
  - `key`
  - `label`
  - `valueType`
  - `required`
  - `description`
  - `defaultValue`
- `WebAigcNodeSchema` 已把节点契约拆成三个明确区域：
  - `inputs`
  - `outputs`
  - `config`
- `server/core/workflow-runtime-engine.ts` 当前已真实消费这些契约字段，例如：
  - 读取节点 `config`
  - 生成等待输入的 `inputSchema`
  - 以 `WebAigcNodeSchema` 作为运行时节点抽象

这说明输入、输出、配置不是松散备注，而是共享领域模型的一部分。

### 4. 定义与 workflow / mission / replay 的挂接关系

结论：可继续保持已完成。

依据如下：

  - `workflowId`
  - `missionId`
  - `sessionId`
  - `replayId`
  - `auditId`
- `shared/workflow-domain.ts` 已在：
  - `WebAigcGraphDefinition.links`

- `shared/mission/projection.ts` 已定义并归并 `MissionProjectionLinks`，用于把 mission 侧与 workflow/runtime 侧链接对齐。
- `server/core/workflow-runtime-engine.ts` 在构建 definition 与 instance 时，已经把 `workflowId / missionId / sessionId / replayId / auditId` 写入 links。
- `server/tests/workflow-graph-projection.test.ts` 已验证 workflow、mission、session 等关联信息可以投影到图实例快照。

因此，这一项不只是“概念上打算挂接”，而是已经有共享链接模型和消费方实现。

### 5. 输出后续 52 个节点 spec 的统一模板约束

  - 勾选规则

这一项的产物更偏“规范文档”而不是“运行时代码”，但从任务表述看，确实已经有明确交付物。

## 当前判断

`web-aigc-platform-domain-model` 这个 spec 当前更像“平台总契约底座”，而不是“某一个节点能力是否跑通”的实现型 spec。

因此它的验收重点应放在：

- 是否形成统一共享类型
- 是否形成统一状态映射
- 是否形成统一链接模型
- 是否形成后续 52 个节点 spec 的模板约束

按这个口径核查，`tasks.md` 当前 5 项都仍然成立。

## 仍需注意的边界

- 当前已经有统一的 `auditId`、`replayId` 等链接字段，但“每条运行时事件都如何统一写入这些系统”属于后续运行时与可观测 spec 的范围，不应在本 spec 里混淆。
- 当前模板约束文档位于 `.kiro/steering`，说明它是平台级规范产物，而不是 `shared/*.ts` 类型契约本身；但从“是否已有统一模板约束产物”这个任务目标来看，仍然满足要求。
- 当前领域模型已经能承载 `selection / confirm_judge / end` 等新运行时能力，但这些节点的执行闭环是否全部完成，属于 node spec / runtime spec 的验收范围，不影响本 spec 的完成判断。

## 最终建议

本轮不需要调整 `tasks.md` 勾选状态，建议保留现状。

如果后续要进一步增强这个 spec，可以补的不是“取消已完成”，而是继续新增更高阶的配套材料，例如：

1. 领域模型版本演进约束
2. 兼容旧 `web-aigc` 编排 JSON 的迁移映射表
3. 统一字段字典与命名保留字清单

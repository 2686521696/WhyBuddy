# Requirements Document

## Introduction

The Autopilot page (SPEC-FIRST blueprint cockpit) provides a language switch (`AutopilotLanguageSwitch`) allowing users to toggle between `zh-CN` and `en-US`. When the user selects Chinese (`zh-CN`), many UI labels and LLM-generated content continue to display in English, breaking the i18n consistency contract. This spec addresses two categories of untranslated content: (1) frontend UI labels that are hardcoded in English or missing Chinese translations, and (2) LLM-generated content that ignores the user's locale because the backend prompts lack locale-aware instructions and the frontend does not pass the `locale` parameter to all relevant API calls.

## Glossary

- **Autopilot_Page**: The SPEC-FIRST blueprint cockpit page located at `client/src/pages/autopilot/`, including its right-rail components, stage panels, and workbench areas.
- **Locale**: The user's selected language preference, typed as `AppLocale = "zh-CN" | "en-US"`, stored in the Zustand global store and persisted to localStorage.
- **RoleStatusStrip**: A horizontal badge strip in the right rail that displays role identifiers (e.g., `intake-analyst`, `repo-researcher`) and their current phase.
- **SpecTreeWorkbench**: The interactive spec tree panel in the right rail that renders node names derived from LLM-generated blueprint data.
- **WorkbenchStatusBar**: A status indicator bar within the workbench area that displays operational labels and state text.
- **ProcessArtifactSplitPanel**: The split-panel component in the right rail that renders lane titles and artifact type labels for the stage process view.
- **Route_Card**: A UI card component that displays a route name and description generated by the LLM routeset planner.
- **Blueprint_API**: The backend API surface under `/api/blueprint/` responsible for triggering LLM generation of routes, spec trees, spec documents, and clarification questions.
- **t_Helper**: The inline i18n helper function `t(locale, zhText, enText)` used throughout the project to select locale-appropriate text.
- **copyDynamic**: The existing translation helper in `copy-dynamic.ts` that maps known English LLM output strings to Chinese equivalents via a static dictionary with regex fallback.
- **Locale_Propagation**: The mechanism by which the frontend passes the current `locale` value to backend API calls so that LLM prompts can be generated in the correct language.

## Requirements

### Requirement 1: RoleStatusStrip Locale-Aware Display

**User Story:** As a Chinese-speaking user, I want role names in the RoleStatusStrip to display in Chinese when my locale is `zh-CN`, so that I can understand which roles are active without reading English identifiers.

#### Acceptance Criteria

1. WHEN the locale is `zh-CN`, THE RoleStatusStrip SHALL display each role identifier using a Chinese label from a role-name mapping dictionary.
2. WHEN the locale is `en-US`, THE RoleStatusStrip SHALL display each role identifier using a human-readable English label (e.g., `Intake Analyst` instead of `intake-analyst`).
3. WHEN a role identifier has no entry in the mapping dictionary, THE RoleStatusStrip SHALL fall back to displaying the raw role identifier unchanged.
4. THE RoleStatusStrip SHALL accept the current locale from the Zustand store without requiring prop drilling from parent components.

### Requirement 2: SpecTreeWorkbench Node Name Localization

**User Story:** As a Chinese-speaking user, I want spec tree node names to display in Chinese when my locale is `zh-CN`, so that I can navigate the spec tree without needing to read English.

#### Acceptance Criteria

1. WHEN the locale is `zh-CN` and a spec tree node name has a Chinese translation available in the existing blueprintCopy dictionary, THE SpecTreeWorkbench SHALL display the Chinese translation via `blueprintCopy(node.title, locale)`.
2. WHEN the locale is `zh-CN` and a spec tree node name has no blueprintCopy dictionary entry, THE SpecTreeWorkbench SHALL display the original LLM-generated text unchanged.
3. WHEN the locale is `en-US`, THE SpecTreeWorkbench SHALL display the original LLM-generated node name unchanged.
4. WHEN the backend generates spec tree node names with locale-aware prompts (Requirement 7), THE SpecTreeWorkbench SHALL display the LLM-generated localized text directly without dictionary lookup.

### Requirement 3: Route Card Locale-Aware Display

**User Story:** As a Chinese-speaking user, I want route names and descriptions on route cards to display in Chinese when my locale is `zh-CN`, so that I can understand the available execution paths.

#### Acceptance Criteria

1. WHEN the locale is `zh-CN` and a route name has a Chinese translation in the copyDynamic dictionary, THE Route_Card SHALL display the Chinese translation for the route name.
2. WHEN the locale is `zh-CN` and a route description has a Chinese translation in the copyDynamic dictionary, THE Route_Card SHALL display the Chinese translation for the route description.
3. WHEN the locale is `zh-CN` and no dictionary entry exists for a route name or description, THE Route_Card SHALL display the original text unchanged.
4. WHEN the locale is `en-US`, THE Route_Card SHALL display the original LLM-generated route name and description unchanged.

### Requirement 4: ProcessArtifactSplitPanel Label Localization

**User Story:** As a Chinese-speaking user, I want lane titles and artifact type labels in the ProcessArtifactSplitPanel to display in Chinese when my locale is `zh-CN`, so that I can understand the stage process layout.

#### Acceptance Criteria

1. WHEN the locale is `zh-CN`, THE ProcessArtifactSplitPanel SHALL display all lane titles using the t_Helper with Chinese text.
2. WHEN the locale is `zh-CN`, THE ProcessArtifactSplitPanel SHALL display all artifact type labels using the t_Helper with Chinese text.
3. WHEN the locale is `en-US`, THE ProcessArtifactSplitPanel SHALL display lane titles and artifact type labels in English.
4. THE ProcessArtifactSplitPanel SHALL read the locale from the Zustand store or receive it via the existing right-rail props context.

### Requirement 5: WorkbenchStatusBar and Stage Label Localization

**User Story:** As a Chinese-speaking user, I want status bar labels and stage labels in the right rail to display in Chinese when my locale is `zh-CN`, so that the entire Autopilot interface feels consistently localized.

#### Acceptance Criteria

1. WHEN the locale is `zh-CN`, THE WorkbenchStatusBar SHALL display all static labels (status text, section titles) using the t_Helper with Chinese text.
2. WHEN the locale is `zh-CN`, THE Autopilot_Page SHALL display stage labels and sub-stage labels in the right rail using the t_Helper with Chinese text.
3. WHEN the locale is `en-US`, THE WorkbenchStatusBar and stage labels SHALL display in English.
4. THE RoleStatusStrip stage label (e.g., `Stage 0`, `Stage 1`) SHALL display as a localized label (e.g., `阶段 0`, `阶段 1`) when the locale is `zh-CN`.

### Requirement 6: Frontend Locale Propagation to Blueprint API

**User Story:** As a developer, I want the frontend to pass the current locale to all Blueprint API calls, so that the backend can instruct the LLM to generate content in the user's preferred language.

#### Acceptance Criteria

1. WHEN the Autopilot_Page initiates a blueprint generation request (job creation, routeset generation, spec tree generation, spec document generation), THE Autopilot_Page SHALL include the current locale value in the API request payload.
2. THE Locale_Propagation SHALL use a `locale` field of type `"zh-CN" | "en-US"` in the request body sent to the Blueprint_API.
3. IF the locale field is missing from the request payload, THEN THE Blueprint_API SHALL default to `"en-US"` for backward compatibility.
4. THE Autopilot_Page SHALL read the locale from the Zustand store at the time of API call initiation, ensuring the most recent user selection is used.

### Requirement 7: Backend LLM Prompt Locale Awareness

**User Story:** As a Chinese-speaking user, I want the LLM to generate route names, node names, route descriptions, and clarification questions in Chinese when my locale is `zh-CN`, so that I do not need to rely on frontend dictionary lookups for translation.

#### Acceptance Criteria

1. WHEN the locale in the request payload is `zh-CN`, THE Blueprint_API routeset prompt builder SHALL include a Chinese-language system instruction directing the LLM to generate route names and descriptions in Chinese.
2. WHEN the locale in the request payload is `zh-CN`, THE Blueprint_API spec tree prompt builder SHALL include a Chinese-language system instruction directing the LLM to generate node names in Chinese.
3. WHEN the locale in the request payload is `zh-CN`, THE Blueprint_API clarification prompt builder SHALL generate clarification questions in Chinese.
4. WHEN the locale in the request payload is `en-US`, THE Blueprint_API prompt builders SHALL generate all content in English.
5. THE Blueprint_API SHALL pass the locale value from the request payload to all prompt builder functions without modification.

### Requirement 8: copyDynamic Dictionary Expansion

**User Story:** As a developer, I want the copyDynamic dictionary to cover all known LLM-generated English strings that appear on the Autopilot page, so that existing content generated before locale propagation is implemented can still display in Chinese.

#### Acceptance Criteria

1. THE copyDynamic dictionary SHALL include Chinese translations for all route names currently generated by the routeset planner (e.g., `PrimaryruntimePath`, `ConservativeFallbackPath`).
2. THE copyDynamic dictionary SHALL include Chinese translations for all known stage sub-step labels generated by the LLM (e.g., `Establish repository-backed executor bridge`, `Select executor-backed runtime path`).
3. WHEN a new LLM-generated string is identified as untranslated in the `zh-CN` locale, THE copyDynamic dictionary SHALL be extended with the corresponding Chinese translation.
4. THE copyDynamic function SHALL continue to return the original English text unchanged when no dictionary match is found, preserving graceful degradation.

### Requirement 9: Clarification Question Localization

**User Story:** As a Chinese-speaking user, I want clarification questions displayed during the blueprint process to appear in Chinese when my locale is `zh-CN`, so that I can understand and respond to them without language barriers.

#### Acceptance Criteria

1. WHEN the locale is `zh-CN` and the backend generates clarification questions with locale-aware prompts, THE Autopilot_Page SHALL display the LLM-generated Chinese clarification questions directly.
2. WHEN the locale is `zh-CN` and a clarification question was generated without locale awareness (legacy data), THE Autopilot_Page SHALL attempt to translate the question using the copyDynamic helper.
3. WHEN the locale is `en-US`, THE Autopilot_Page SHALL display clarification questions in English unchanged.

### Requirement 10: No Regression on Excluded Scope

**User Story:** As a developer, I want to ensure that the i18n consistency changes do not affect content that should remain in English, so that technical outputs and 3D scene labels are not inadvertently translated.

#### Acceptance Criteria

1. THE Autopilot_Page SHALL NOT translate console log output or debug messages regardless of locale.
2. THE Autopilot_Page SHALL NOT translate API endpoint paths or technical identifiers (e.g., `POST /api/blueprint/jobs`) regardless of locale.
3. THE 3D Scene (Scene3D) labels SHALL NOT be affected by this spec and SHALL remain unchanged.
4. THE existing `t_Helper` function signature `t(locale, zhText, enText)` SHALL remain unchanged to maintain backward compatibility with all existing consumers.

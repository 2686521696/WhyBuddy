# `web-aigc-node-open_page` 现状核查

## 核查结论

当前主仓已经具备 `open_page` 的最小可运行闭环，且本轮收口后可按 `4/4` 视为完成：

1. 已有独立节点执行适配器，能够返回前端可消费的页面打开目标描述。
2. 已支持 `params / query / context` 透传，其中：
   - `params` 用于路由占位符替换；
   - `query` 用于构造最终 `href`；
   - `context` 原样保留给宿主前端壳层使用。
3. 已支持可选权限校验；若接入 `permissionEngine`，则会基于 `api:call` 拒绝无权限的页面打开动作。
4. 已有节点级测试和路由级测试，覆盖成功、拒绝、输入缺失等关键路径。

## 代码事实

### 1. 节点执行入口

文件：
- `server/routes/node-adapters/open-page-node-adapter.ts`

当前实现能力：
- 支持节点类型 `open_page`
- 支持目标类型：
  - `internal_route`
  - `external_url`
  - `task_detail`
- 支持打开模式：
  - `push`
  - `replace`
  - `new_tab`

返回结构包含：
- `target.kind`
- `target.pageId`
- `target.href`
- `target.route`
- `target.params`
- `target.query`
- `target.title`
- `target.openMode`
- `payload.params`
- `payload.query`
- `payload.context`

这已经满足“返回可打开页面目标 + 参数透传 + 前端可消费 target 描述”的最小要求。

### 2. 权限校验

文件：
- `server/routes/node-adapters/open-page-node-adapter.ts`
- `server/index.ts`

当前实现方式：
- `open_page` adapter 支持注入 `permissionEngine`
- 校验资源类型固定为 `api`
- 操作为 `call`
- 资源字符串以 `POST /api/open-page/nodes/execute:...` 形式描述本次打开动作

若校验失败：
- 返回 `ok: false`
- 返回 `output.status = "denied"`
- 返回 `governance.permission.reason / suggestion`

这满足 spec 中“治理控制”的最小要求。

### 3. 路由入口

文件：
- `server/routes/open-page.ts`
- `server/index.ts`

当前提供的服务端入口：
- `POST /api/open-page/nodes/execute`

行为：
- `nodeType !== open_page` 时返回 `400`
- 权限拒绝时返回 `403`
- 合法请求返回 `200`，并返回标准化 `target + payload`

### 4. 测试证据

文件：
- `server/tests/open-page-node-adapter.test.ts`
- `server/tests/open-page-routes.test.ts`

已覆盖场景：
- 内部路由目标生成
- 外部页面目标生成
- `params / query / context` 透传
- 权限拒绝
- 缺少目标定义时报错
- 路由层的 `400 / 403 / 200` 状态映射

## 当前边界

本轮仅实现“页面打开描述能力”，未实现以下内容，这些也不在当前 spec 的最小闭环范围内：

- 不做真实浏览器跳转组件
- 不做前端 UI 壳层的实际路由跳转执行
- 不做页面存在性探测
- 不做复杂页面注册中心

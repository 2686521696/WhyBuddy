# 需求文档：本地 UE 渲染运行时

## 简介

本规格定义本地 UE5 实例的启动、画面输出与浏览器接入方式。核心目标是在开发机上运行 UE5 渲染进程，并通过本地 WebRTC / Pixel Streaming 本地模式将画面推送到前端浏览器，形成"本地 GPU 渲染 + 浏览器消费"的最小闭环。这不是云端推流服务器，而是面向单机开发与演示的本地渲染运行时。

## 需求

### 需求 1：UE5 实例本地启动与生命周期管理

**用户故事：** 作为开发者，我希望通过一条命令启动本地 UE5 渲染实例，以便快速进入开发调试流程。

#### 验收标准

1. 当用户执行启动脚本时，系统应在本地启动 UE5 实例并加载指定场景关卡。
2. 系统应支持通过环境变量或配置文件指定 UE5 安装路径、项目路径、启动参数与渲染分辨率。
3. 系统应在 UE5 实例启动后 30 秒内完成场景加载并进入可渲染状态。
4. 系统应提供停止、重启命令，并在进程异常退出时输出错误日志。

### 需求 2：本地 Pixel Streaming 画面输出

**用户故事：** 作为前端开发者，我希望 UE5 渲染画面能通过本地网络推送到浏览器，以便在前端页面中实时查看 3D 场景。

#### 验收标准

1. 系统应启用 UE5 内置 Pixel Streaming 插件，在本地端口（默认 8888）提供 WebRTC 信令服务。
2. 系统应支持本地回环地址（127.0.0.1）与局域网地址两种接入模式。
3. 画面输出延迟应控制在 100ms 以内（本地回环场景）。
4. 系统应支持动态调整输出分辨率（720p / 1080p / 自适应）。

### 需求 3：会话连接与断线重连

**用户故事：** 作为用户，我希望浏览器与 UE 渲染实例之间的连接在断开后能自动恢复，以便不中断工作流。

#### 验收标准

1. 当浏览器端检测到 WebRTC 连接断开时，系统应在 5 秒内自动发起重连。
2. 系统应支持最多 3 次自动重连尝试，每次间隔递增（1s / 3s / 5s）。
3. 若重连失败，系统应通知前端进入降级模式（回退到 Three.js 或预渲染视频）。
4. 重连成功后，系统应恢复到断开前的场景状态。

### 需求 4：调试模式与开发工具支持

**用户故事：** 作为开发者，我希望在调试模式下能查看渲染帧率、网络延迟和 GPU 占用，以便定位性能瓶颈。

#### 验收标准

1. 系统应提供调试模式启动参数，启用后在浏览器端叠加 FPS、延迟、分辨率等 HUD 信息。
2. 系统应将 UE5 实例的 GPU 占用、显存使用、渲染帧率通过 WebSocket 推送到前端。
3. 调试模式应支持热切换，无需重启 UE5 实例。

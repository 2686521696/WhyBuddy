# web-aigc-node-start 现状核查

## 结论

基于仓库当前实现，`web-aigc-node-start` 的 4 个任务项都已经具备回写条件，可以勾选。

本次判断依据来自：

- `shared/workflow-input.ts` 的输入模型与归一化能力
- `server/routes/workflows.ts` 的入口接收、去重、上下文初始化与 mission 链接逻辑
- `server/core/workflow-engine.ts` 的启动落库与初始上下文写入
- `server/core/workflow-graph-projection.ts`、`server/tasks/mission-projection.ts`、`server/core/aigc-monitoring-projection.ts` 的后续投影消费
- 相关路由与单元测试

本次未修改任何主线实现代码。

## 任务项逐条判断

### 1. 定义开始节点输入模型

结论：已完成，可勾。

依据：

- `shared/workflow-input.ts`
  - 已定义 `WorkflowInputAttachment`
  - 已定义 `WorkflowInputProjectionContext`
  - 已提供 `normalizeWorkflowAttachment()` / `normalizeWorkflowAttachments()`
  - 已提供 `normalizeWorkflowInputProjection()`
  - 已提供 `buildWorkflowInputSignature()`
  - 已提供 `buildWorkflowDirectiveContext()`
- `server/routes/workflows.ts`
  - `POST /api/workflows` 已接收 `directive`
  - 已接收并归一化 `attachments`
  - 已接收并归一化 `projection/sessionId/sourceApp`

这已经满足 spec 中“接收用户输入、附件、全局变量，并输出标准化运行上下文”的最小输入建模部分。现阶段虽然没有单独命名为 `start node input schema` 的类型，但在运行入口上已经形成统一输入模型。

### 2. 定义上下文初始化逻辑

结论：已完成，可勾。

依据：

- `server/routes/workflows.ts`
  - 会基于 `directive + attachments` 生成 `inputSignature`
  - 会基于 `directive + attachments` 生成 `directiveContext`
  - 会调用 `workflowEngine.startWorkflow(normalizedDirective, { attachments, directiveContext, inputSignature })`
- `server/core/workflow-engine.ts`
  - `startWorkflow()` 会再次兜底生成 `attachments / directiveContext / inputSignature`
  - 会将这些输入写入 `workflow.results.input`
  - 后续 `runPipeline()` 不是直接使用原始 `directive`，而是用已经初始化过的 `directiveContext`
- `server/tests/workflow-directive-context.test.ts`
  - 已验证附件、视觉描述等会被注入到指令上下文

这说明开始节点不仅“收到了输入”，还完成了输入标准化、上下文展开和签名生成，属于明确的上下文初始化逻辑。

### 3. 写入启动事件与初始快照

结论：已完成，可勾。

依据：

- 初始快照：
  - `server/core/workflow-engine.ts`
    - 在 `startWorkflow()` 中写入：
      - `status: "running"`
      - `started_at`
      - `results.input.attachments`
      - `results.input.directiveContext`
      - `results.input.signature`
  - `server/routes/workflows.ts`
    - 会补写：
      - `results.input.sessionId`
      - `results.input.sourceApp`
      - `results.input.projection`
- 启动事件：
  - `server/routes/workflows.ts`
    - 创建 mission 后会调用 `missionRuntime.markMissionRunning(mission.id, "execute", \`Workflow ${workflowId} started\`)`
  - `server/tasks/mission-store.ts`
    - `markRunning()` 会写入 `task.events.push({ type: "progress", message: detail || \`Running ${stageKey}\`, ... })`
    - 同时推进 mission/stage 到 `running`
- 路由测试：
  - `server/tests/workflows-routes.test.ts`
    - 已验证 workflow 投影输入会被写入并与 mission 建立链接

严格来说，当前实现没有单独定义一个名为 `workflow_started` 的专用事件类型；但从 spec 的最小闭环要求看，已经具备：

- workflow 启动时间戳
- workflow 初始输入快照
- mission running 事件记录

因此可以认定该项已满足。

### 4. 验证与后续节点的上下文传递

结论：已完成，可勾。

依据：

- `server/core/workflow-engine.ts`
  - `runPipeline()` 直接使用开始阶段生成的 `directiveContext`
  - `getWorkflowDirectiveContext()` 会从 `workflow.results.input.directiveContext` 回读上下文
  - 在反馈阶段继续把“原始用户指令/上下文”传入后续 agent 调用
- `server/core/workflow-graph-projection.ts`
  - graph snapshot 会保留 `workflow.id / missionId / sessionId / startedAt`
  - 对 controlflow 路径会优先读取 `definition.input.directive` 或 `workflow.results.input.directiveContext`
- `server/tasks/mission-projection.ts`
  - mission projection 会回读 workflow input
  - 会对外暴露：
    - `attachmentCount`
    - `inputSignature`
    - `sourceApp`
    - `sessionId`
- `server/core/aigc-monitoring-projection.ts`
  - monitoring 侧会优先读取 `workflow.results.input.projection`
- 测试证据：
  - `server/tests/workflows-routes.test.ts`
    - 已验证 workflow projection input 会写入并创建 mission projection links
  - `server/tests/mission-routes.test.ts`
    - 已验证 `/api/tasks/:id/projection` 返回 mission projection view
  - `server/tests/aigc-monitoring-routes.test.ts`
    - 已验证 monitoring session detail 会优先使用 projection `sessionId/sourceApp`
  - `server/tests/workflow-graph-projection.test.ts`
    - 已验证 graph projection 会带出 `sessionId`

这说明开始节点产出的上下文不只是落在数据库里，而是已经进入 graph/projection/monitoring 等后续消费链路。

## 仍需补强的地方

虽然任务项可以勾选，但仍有两点值得后续补强：

- `start` 节点目前更多是“入口路由 + workflow/mission 初始化约定”，而不是一个在 runtime 图定义中显式统一暴露的独立 node adapter。
- “启动事件”当前主要依赖 `started_at` 与 mission progress event，没有看到单独命名的 `workflow_started` 专用事件类型；若后续平台要做更细粒度 observability，对事件语义可以再收紧。

这些属于增强项，不影响本次 `tasks.md` 的回写。

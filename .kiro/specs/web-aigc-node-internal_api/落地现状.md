# 内部接口节点落地现状

## 一、已存在的可复用实现

仓库内已经具备 `internal_api` 节点的第一层真实承接能力，并非从零开始：

- `server/tool/api/internal-api-adapter.ts`
  - 负责内部目标目录、参数读取、结果标准化与具体执行分发。
- `server/tool/api/auto-agent-adapter.ts`
  - 已支持 `kind: "internal_api"`，可将内部接口调用收敛到统一执行协议。
- `server/routes/a2a.ts`
  - `/api/a2a/auto-agent` 已对外接受 `internal_api` 类型请求。

当前已沉淀的内部目标包括：

- `mission.projection.get`
- `mission.session.get`
- `workflow.graph_instance_snapshot`
- `aigc_monitoring.instances`
- `aigc_monitoring.instance_detail`
- `aigc_monitoring.session_detail`
- `web_aigc.risk_action_catalog`

这意味着 `internal_api` 节点已经具备“以 `targetId` 方式调用内部能力簇”的薄适配基础，可以先作为图运行时里的高耦合企业节点承接层。

## 二、与 spec 三项核心诉求的映射关系

### 1. 内部调用

当前实现并不是按“服务标识 + 路径”的 HTTP 透传模式，而是采用更薄的一层内部目标映射：

- 通过 `targetId` 选择内部能力目标
- 通过 `workflowId` / `metadata` 读取实例、任务、监控等上下文
- 通过 `InternalApiExecutor.execute()` 执行并返回统一结构

这与当前设计文档中“高耦合企业节点，后期定制适配”的定位一致，适合作为第一阶段落地方案。

### 2. 鉴权治理

当前已经具备 `internal_api` 统一入口级的权限与审计钩子：

- `server/tool/api/internal-api-adapter.ts`
  - 新增了可选注入的 `permissionEngine` 与 `auditLogger`
  - 当启用治理能力时，强制要求 `metadata.agentId` 与 `metadata.token`
  - 统一资源标识为 `internal_api:<targetId>`
  - 在入口执行 `permission + governance` 判断
  - 统一记录 `allowed / denied / error` 审计结果
- 审计元数据已透传：
  - `targetId`
  - `workflowId`
  - `missionId`
  - `stage`
  - `inputPreview`

这说明本节点已经不只是“下游局部能力自带治理”，而是具备了节点级统一治理入口。

### 3. 结果标准化

这一点落地程度较高：

- `internal-api-adapter` 统一返回 `output / targetLabel / operation / response`
- `auto-agent-adapter` 继续包装为统一 `AutoAgentExecutionResult`
- 路由层可以直接暴露标准化结果

对于节点运行时接入而言，结果契约已经足够稳定，可直接复用。

## 三、现有测试覆盖情况

已有测试文件：

- `server/tests/internal-api-adapter.test.ts`
- `server/tests/auto-agent-adapter.test.ts`
- `server/tests/auto-agent-routes.test.ts`

已覆盖的方向：

- mission projection 查询
- mission session 查询
- workflow graph snapshot 查询
- monitoring instance detail 查询
- risk action catalog 查询
- `internal_api` 通过 auto-agent 与 route 的接入链路
- 缺少 `agentId` 时拒绝执行
- 缺少 `token` 时拒绝执行
- 权限拒绝时阻断执行并写入审计
- 成功执行时写入 allowed 审计

未充分覆盖的方向：

- `targetId` 不存在
- 缺失 `missionId` / `workflowId`
- mission/workflow 不存在
- 节点级错误回退策略

## 四、当前缺口与后续最小补齐项

如果后续继续推进代码落地，建议按下面顺序补齐：

1. 为 `internal-api-adapter` 继续补充负向测试，至少覆盖缺参、目标不存在、资源不存在三类错误。
2. 明确节点失败策略：
   - 是否中断工作流
   - 是否返回空输出
   - 是否允许按目标类型降级
3. 如果未来要对接更多内部服务，再评估是否从 `targetId` 目录式映射演进为“服务标识 + 操作名 + 参数模板”的更通用协议。
4. 在现有 agent 级权限基础上，再向租户、多环境、限额等更高阶治理维度扩展。

## 五、结论

`web-aigc-node-internal_api` 这个 spec 目前可以判定为：

- 调用结构：已具备第一阶段落地基础
- 输出结构：已具备第一阶段落地基础
- 权限审计：节点级统一钩子已落地，后续可继续向更高阶治理扩展
- 错误回退：已有基础报错机制，但专项验证未完成

因此，这个节点已经可以按“薄适配、少目标、统一输出、统一治理入口”的方式进入后续图运行时集成；但仍不建议在当前阶段把它表述成“已具备完整企业级治理能力”。

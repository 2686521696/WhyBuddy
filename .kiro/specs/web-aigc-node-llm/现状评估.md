# web-aigc-node-llm 现状评估

## 结论

基于仓库当前实现，`tasks.md` 中剩余两项未勾任务已经具备回写条件：

- `支持结构化输出`
- `写入成本与延迟指标`

本次回写依据的是仓库内已经存在的通用 LLM 抽象、runtime 调用链、telemetry/cost 记录链路，以及对应测试。未新增任何主线实现代码。

## 任务项与证据

### 1. 支持结构化输出

已具备可复用的结构化输出能力，且不是停留在接口层，而是已经接入到 runtime 调用路径。

关键证据：

- `shared/workflow-runtime.ts`
  - `LLMProvider` 已定义 `callJson<T>()`
  - `AgentHandle` 已定义 `invokeJson<T>()`
- `shared/runtime-agent.ts`
  - `composeAgentMessages(..., jsonMode = true)` 会注入“仅返回合法 JSON”的系统提示
  - `RuntimeAgent.invokeJson()` 会走 `llmProvider.callJson<T>()`
- `shared/llm/contracts.ts`
  - `LLMGenerateOptions` 已提供 `jsonMode?: boolean`
- `server/tests/workflow-runtime-engine.test.ts`
  - runtime 侧测试桩已经包含 `callJson()`
- `server/tests/agent-vision-events.test.ts`
  - 已覆盖 `invokeJson()` 路径
- `server/tests/nl-command/decision-support.test.ts`
  - 已断言调用参数包含 `jsonMode: true`
- `server/tests/nl-command/command-analyzer.test.ts`
  - 已断言调用参数包含 `jsonMode: true`

判断：

- 需求文档要求“输出应支持文本、JSON 或结构化结果，并纳入监控”。
- 当前实现中，文本路径与 JSON 路径都已经作为通用 LLM/runtime 能力存在，因此该任务可以勾选。

### 2. 写入成本与延迟指标

服务端主调用链已经在 LLM 调用完成或失败时写入延迟、token、成本相关指标，并且这些指标可以通过现有路由读取。

关键证据：

- `shared/llm/contracts.ts`
  - `LLMGenerateResult` 已定义 `usage`、`latencyMs`、`model`、`provider`
- `shared/telemetry/contracts.ts`
  - 已定义 `telemetry:llm_call` 事件，包含 token 与 `latencyMs`
  - 已定义 `cost:llm_call` 事件
- `shared/telemetry.ts`
  - 已提供 `LLMCallRecord`、快照结构与成本估算能力
- `shared/cost.ts`
  - 已提供 pricing table、`estimateCost()`、成本快照与记录结构
- `server/core/llm-client.ts`
  - `callProvider()` 成功或失败都会写入 `telemetryStore.recordLLMCall(...)`
  - `callLLM()` 成功或失败都会写入 `costTracker.recordCall(...)`
  - 记录中已包含 `tokensIn`、`tokensOut`、`durationMs`、`actualCost`
- `server/routes/telemetry.ts`
  - 已暴露 `/api/telemetry/live`、`/api/telemetry/history`
- `server/routes/cost.ts`
  - 已暴露 `/api/cost/live`、`/api/cost/history`、`/api/cost/budget`
- `server/tests/telemetry.test.ts`
  - 已验证 LLM call 记录会汇总到 telemetry snapshot
- `server/tests/cost-api.test.ts`
  - 已验证 cost 路由可以返回累计 token/cost/call 指标
- `server/tests/cost-tracker.test.ts`
  - 已覆盖 `recordCall()` 累计、聚合与阈值逻辑
- `server/tests/cost-observability.test.ts`
  - 已覆盖 `estimateCost()` 与成本记录保持性

判断：

- 从“写入”角度看，服务端主路径已经完成成本与延迟指标的落库/聚合写入。
- 从“可观测”角度看，telemetry 与 cost 路由也已可读取这些结果。
- 因此该任务可以勾选。

## 边界与保留说明

虽然上述任务可以勾选，但仍建议保留以下现状说明，避免后续误判为“所有端到端场景都完全一致”：

- `client/src/lib/browser-llm.ts`
  - 浏览器直连路径会记录 `durationMs`
  - 但当前 `tokensIn`、`tokensOut`、`cost` 仍写为 `0`
- `client/src/lib/browser-telemetry-store.ts`
  - 浏览器侧 telemetry 快照会累加这些字段，因此浏览器直连模式下的 token/cost 精度仍偏弱

这意味着：

- 若本 spec 的验收口径是“LLM 节点服务端主链路已经支持结构化输出，并已写入成本/延迟指标”，当前实现已经满足。
- 若后续要把“浏览器直连模式下的精确 token/cost 采集”也纳入同一条验收，则还需要额外补线，但这不影响本次任务项回写。

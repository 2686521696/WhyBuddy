## 任务逐项判定

### 1. 定义通用 LLM 输入输出结构

判定：已完成

依据：

- `shared/workflow-runtime.ts` 已定义：
  - `LLMMessage`
  - `LLMCallOptions`
  - `LLMResponse`
  - `LLMProvider`
- `shared/llm/contracts.ts` 还补充了更完整的多提供商契约：
  - `LLMGenerateOptions`
  - `LLMGenerateResult`
  - `LLMTokenUsage`
  - `ILLMProvider`
- 输入侧已覆盖：
  - 提示消息
  - 模型
  - 温度
  - `maxTokens`
  - `jsonMode`
- 输出侧已覆盖：
  - 文本内容
  - token usage
  - `latencyMs`
  - model/provider 标识

因此，这一项不只是“有一个 call 方法”，而是已经形成了通用 LLM 输入输出模型。

### 2. 接入现有模型抽象

判定：已完成

依据：

- `server/runtime/server-runtime.ts` 已把服务端运行时的 `llmProvider` 统一挂接到：
  - `callLLM()`
  - `callLLMJson()`
- `server/core/llm-client.ts` 已实现多提供商调用、fallback、模型选择、超时与错误处理。
- `shared/runtime-agent.ts` 中：
  - `RuntimeAgent.invoke()` 走 `llmProvider.call()`
  - `RuntimeAgent.invokeJson()` 走 `llmProvider.callJson()`
- `server/tests/llm-client.provider-fallback.test.ts` 已验证 provider fallback 的真实调用链。

因此，`llm` 节点已经不是“尚未落地的概念接口”，而是确实接入了现有模型抽象与服务端调用主链。

### 3. 支持结构化输出

判定：已完成

依据：

- `shared/workflow-runtime.ts` 中 `LLMProvider` 已定义 `callJson<T>()`
- `shared/runtime-agent.ts` 中：
  - `composeAgentMessages(..., jsonMode = true)` 会注入“只返回合法 JSON”的系统提示
  - `invokeJson()` 会走 `llmProvider.callJson<T>()`
- `server/core/llm-client.ts` 中：
  - `callLLMJson()` 会以 `jsonMode: true` 调用 `callLLM()`
  - 并在返回后做 JSON 解析与容错提取
- 测试侧：
  - `server/tests/agent-vision-events.test.ts` 已覆盖 `invokeJson()`
  - `server/tests/nl-command/decision-support.test.ts` 已断言调用参数包含 `jsonMode: true`
  - `server/tests/nl-command/command-analyzer.test.ts` 已断言调用参数包含 `jsonMode: true`
  - `server/tests/llm-client.provider-fallback.test.ts` 说明 `callLLMJson()` 在 provider fallback 场景下仍能返回结构化 JSON

因此，“支持结构化输出”当前已是实际能力，不只是接口名存在。

### 4. 写入成本与延迟指标

判定：已完成

依据：

- `shared/telemetry/contracts.ts` 已定义：
  - `telemetry:llm_call`
  - 包含 `promptTokens / completionTokens / totalTokens / latencyMs`
- `server/core/llm-client.ts` 中：
  - `callProvider()` 成功和失败都会调用 `telemetryStore.recordLLMCall(...)`
  - `callLLM()` 成功和失败都会调用 `costTracker.recordCall(...)`
  - 记录内容包含：
    - token 输入输出
    - `durationMs`
    - `actualCost`
    - model
    - agentId / missionId / sessionId
- `server/routes/telemetry.ts` 已暴露：
  - `/api/telemetry/live`
  - `/api/telemetry/history`
- `server/routes/cost.ts` 已暴露：
  - `/api/cost/live`
  - `/api/cost/history`
  - `/api/cost/budget`
- 测试侧：
  - `server/tests/telemetry.test.ts` 已验证 LLM 调用记录会汇总进 telemetry snapshot
  - `server/tests/cost-tracker.test.ts` 已验证 `recordCall()` 的累计、聚合与阈值逻辑
  - `server/tests/cost-api.test.ts` 已验证 cost 路由可返回累计 token/cost/call 指标

因此，这一项当前已经不是“计划接入”，而是服务端主调用链已实际写入并可被读取。

## 边界说明

虽然本 spec 当前 4 项都可保持已完成，但仍有一个应明确的边界：

- 本次结论主要建立在服务端主链路与共享 runtime 抽象之上。
- 浏览器直连模式下的 token/cost 精度如果单独拉高验收口径，仍可能需要额外补线。

但这不影响当前 `web-aigc-node-llm` spec 的 4 个任务项继续保持已完成状态。

## 当前结论

`web-aigc-node-llm` 当前已经具备比较完整的通用节点底座能力：

- 有统一输入输出契约
- 有服务端主调用抽象
- 有结构化输出链路
- 有成本与延迟观测链路

按当前仓库事实，这份 spec 可以继续维持“4 项全部完成”的状态。

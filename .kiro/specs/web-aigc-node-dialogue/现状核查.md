# web-aigc-node-dialogue 现状核查

## 结论

基于 2026-04-22 当前主仓代码事实，`web-aigc-node-dialogue` 现在可以确认完成 4 项中的 3 项：

- 已完成：`定义对话节点输入输出结构`
- 已完成：`对接聊天与会话记录`
- 未完成：`接入检索与工具增强`
- 已完成：`打通实例关联会话查看`

本次推进把 `dialogue` 节点从“只会调 LLM 的节点执行入口”推进到了“能落 workflow message、能落 session exchange、能把监控所需元数据回写出来”的最小闭环。

仍然保留未完成的是第 3 项，因为当前只是支持把 `context / citations / toolCalls / thinking` 作为增强信息透传到模型与监控元数据，并没有形成主动调用检索服务或工具执行器的编排链路。

## 本次实际补齐了什么

### 1. 节点级会话记录闭环

`dialogue` 节点执行后，现在会在满足关联字段时写入：

- `workflow message`
  - 用户消息写入一条 `workflow-user -> dialogue-agent`
  - 助手回复写入一条 `dialogue-agent -> workflow-user`
- `session exchange`
  - 当传入 `agentId` 时，会调用 `sessionStore.appendLLMExchange(...)`

对应实现位于：

- `server/routes/node-adapters/chat-node-adapter.ts`

这意味着 `dialogue` 节点不再只是临时返回响应体，而是能把对话结果沉淀到平台已有的消息与会话存储里。

### 2. 监控元数据回写

当前 `dialogue` 节点已支持接收并回写以下增强字段：

- `citations`
- `toolCalls`
- `thinking`
- `workflowId`
- `sessionId`
- `missionId`
- `agentId`
- `stage`

这些字段会落到 assistant message 的 `metadata` 中，也会写入 session exchange 的 `metadata` 中。

由于 `server/core/aigc-monitoring-projection.ts` 本来就会从 workflow message 的 `metadata` 读取：

- `thinking`
- `citations`
- `toolCalls`

因此本次补齐后，`dialogue` 节点产出的消息已经能直接被实例会话监控视图消费。

### 3. 薄接线增强信息注入

当前 `dialogue` 节点会把以下上游增强信息整理进 system message：

- `context`
- `variables`
- `citations`
- `toolCalls`

这说明：

- 上游检索结果、工具结果、人工输入，已经可以通过统一输入结构送进 `dialogue` 节点
- 节点输出后，这些增强信息也能在监控视图中看到

但这还不等于：

- `dialogue` 节点会主动发起 RAG 检索
- `dialogue` 节点会主动调用工具执行器

所以这一块只能算“增强信息薄接线已补上”，不能算“检索与工具增强完整接入”。

  - 已实现 `buildMessages()`
  - 已实现 `executeChatNode()`
- `server/routes/chat.ts`
  - 已暴露 `POST /api/chat/nodes/execute`
  - 已接受 `nodeType = "dialogue"`
- `server/tests/chat-routes.test.ts`
  - 已覆盖 `dialogue` 执行入口
  - 已验证缺少 `prompt/messages` 时会返回 400

补充说明：

- “多轮”当前仍主要体现在调用方可显式传入 `messages`
- 节点不会自动从平台 session 中反查历史消息并组装输入

### 2. 对接聊天与会话记录

结论：已完成，可勾。

依据：

- `server/routes/node-adapters/chat-node-adapter.ts`
  - `dialogue` 节点新增支持 `workflowId / sessionId / missionId / agentId / stage`
  - 执行后会写入 `messageStore.createMessage(...)`
  - 执行后会写入 `sessionStore.appendLLMExchange(...)`
- `server/tests/chat-routes.test.ts`
  - 已验证 `dialogue` 节点执行后会写入 2 条 workflow message
  - 已验证 `dialogue` 节点执行后会写入 1 组 session exchange

当前闭环口径已经成立：

- 有节点执行入口
- 有 LLM 响应
- 有会话持久化
- 有测试证据

### 3. 接入检索与工具增强

结论：暂不建议勾选。

当前已经具备的事实：

- `dialogue` 节点支持接收 `context`
- `dialogue` 节点支持接收 `citations`
- `dialogue` 节点支持接收 `toolCalls`
- 这些增强信息会进入 system message
- 这些增强信息会回写到 workflow/session metadata

但仍缺少以下关键证据：

- `dialogue` 节点主动调用 RAG 检索入口
- `dialogue` 节点主动调用工具执行器
- `dialogue` 节点对工具失败、重试、工具选择等编排逻辑负责

因此当前更准确的描述是：

- 已支持“增强信息透传”
- 尚未完成“检索与工具增强链路接入”

### 4. 打通实例关联会话查看

结论：已完成，可勾。

依据：

- `server/tasks/mission-projection.ts`
  - mission session 视图依赖 `db.getMessagesByWorkflow(...)`
  - 也会读取 `sessionStore.getMissionWorkflowEntries(...)`
- `server/core/aigc-monitoring-projection.ts`
  - 已能从 workflow message `metadata` 读取 `thinking / citations / toolCalls`
- `server/tests/chat-routes.test.ts`
  - 已用 `buildMonitoringSessionDetail(...)` 验证 `dialogue` 节点写入的 assistant message 可被监控投影读出
  - 已验证 `thinking / citations / toolCalls` 会进入监控会话消息

当前虽然没有单独新增 mission route 测试，但从平台实际读取路径看：

- `dialogue` 节点已经把消息写进了 mission / monitoring 视图所依赖的存储
- 监控投影也已经读到了这些消息和增强元数据

因此“实例关联会话查看”这项已经具备节点级闭环证据。

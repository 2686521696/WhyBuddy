# web-aigc-node-dialogue 现状核查

## 核查结论

基于 2026-04-23 当前主仓代码事实，`web-aigc-node-dialogue` 可以判定为已完成最小闭环，`tasks.md` 中的 4 项任务勾选成立：

- 已完成：定义对话节点输入输出结构
- 已完成：对接聊天与会话记录
- 已完成：接入检索与工具增强
- 已完成：打通实例关联会话查看

本轮补强后，`dialogue` 与 `document_search` 的联动证据更完整，能够覆盖以下真实运行路径：

1. 调用方显式传入 `prompt` 或 `messages`
2. 若配置 `documentSearch`，节点优先使用 `documentSearch.query`，否则回退到 `prompt`，再回退到最新一条用户消息
3. 检索结果被归并为 `context`、`citations`、`toolCalls`
4. 上游传入的 `citations / toolCalls / thinking` 会与检索增强结果合并后再写回
5. route 级注入的 `documentSearch executor` 能直接下沉到 `/api/chat/nodes/execute` 的节点执行链路
6. runtime built-in `dialogue` adapter 也可通过注入的 `documentSearch executor` 走主线检索能力，而不需要额外改热路径主文件
7. 结果被落入 `workflow message` 与 `session exchange`
8. `workflowId / thinking / citations / toolCalls` 会按统一口径进入输出与留痕元数据
9. 结构化 `toolCalls.result` 会被序列化保留，而不是在写回时丢失
10. 监控链路可消费 `thinking / citations / toolCalls`

## 代码事实

### 1. 对话节点执行入口已经稳定

- `server/routes/chat.ts` 提供 `POST /api/chat/nodes/execute`
- `nodeType` 支持 `llm` 与 `dialogue`
- 当 `prompt` 与 `messages` 同时缺失时会返回 400

对应文件：

- `server/routes/chat.ts`
- `server/routes/node-adapters/chat-node-adapter.ts`
- `server/tests/chat-routes.test.ts`

### 2. dialogue 已具备 document_search 主动增强能力

`dialogue` 节点可在执行前主动发起一次 `document_search`，并将检索结果折叠进系统提示与可观测元数据：

- 归并到 `context`
- 归并到 `citations`
- 归并到 `toolCalls`

本轮新增核对点：

- 当没有显式 `prompt` 时，可从 `messages` 中最新的用户消息回退提取检索查询词
- 上游已有 `citations / toolCalls` 不会被覆盖，而是与检索结果合并
- 合并后的增强元数据会同时写入 workflow message 和 session exchange
- route 级 `documentSearch` 注入已验证可直接驱动 `dialogue` 节点执行，不依赖默认检索器
- `workflowId` 现已进入用户消息、助手消息与 session exchange 的统一元数据
- 结构化 `toolCalls.result` 现已在输出、workflow message metadata、session exchange metadata 中保持一致
- runtime built-in `dialogue` adapter 已验证能够消费注入的 `documentSearch executor`，并把检索增强结果传入 LLM 系统消息与实例输出

对应文件：

- `server/routes/node-adapters/chat-node-adapter.ts`
- `server/tests/chat-node-document-search.test.ts`
- `server/tests/workflow-runtime-engine.test.ts`

### 3. 会话落库与监控投影链路成立

当前 `dialogue` 节点执行后已经可以形成会话闭环：

- 用户输入写入 `workflow message`
- 助手输出写入 `workflow message`
- 当存在 `agentId` 时，追加写入 `session exchange`
- `workflowId / thinking / citations / toolCalls` 进入 message metadata，可供监控投影读取
- `session exchange` 复用与助手消息一致的增强元数据
- runtime built-in adapter 路径也满足同样的留痕要求，不只是 route 层能做到

对应文件：

- `server/routes/node-adapters/chat-node-adapter.ts`
- `server/tests/chat-routes.test.ts`
- `server/tests/workflow-runtime-engine.test.ts`

### 4. knowledge 与 dialogue 的关系是“上游增强兼容”，不是“内嵌同一路由”

当前仓库中，`knowledge_qa / qa_search` 是独立的知识节点链路，不是 `dialogue` 路由内部直接调用的固定分支。但其输出结构已经可以作为 `dialogue` 的上游增强输入使用。

本轮补强后，`qa_search` 侧的测试明确验证了这些面向对话增强的事实：

- 有稳定的 `citations`
- 有可直接注入上下文的 `context`
- 有可展示的 `evidenceList`
- 有可排序的 `matches`

对应文件：

- `server/routes/knowledge.ts`
- `server/routes/node-adapters/knowledge-node-adapter.ts`
- `server/tests/knowledge-routes.test.ts`

## 当前边界

以下能力已被纳入当前完成口径：

- 显式传入多轮消息的对话执行
- 一次 `document_search` 主动增强
- workflow/session 级别的结果留痕
- 监控读取 `thinking / citations / toolCalls`

以下能力仍不应算作本 spec 已完成：

- 节点自动回查完整 session 历史并自行组装消息
- 多工具自主选择
- 失败重试与审批治理
- 完整的多工具编排平台能力

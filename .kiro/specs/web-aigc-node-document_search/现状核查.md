# 文档检索节点现状核查

## 核查结论

`web-aigc-node-document_search` 已具备一条可用的最小闭环：调用方可以通过 `POST /api/rag/web-aigc/document-search` 提交检索请求，服务端会将请求映射到现有 RAG 检索能力，并返回按文档聚合的命中证据。按当前仓库实现判断，4 个任务项现在都可以勾选，其中第 4 项属于“项目级 collection 权限 + 检索范围过滤”的最小权限闭环。

本轮补充核查后，可以进一步确认三点：

- `projectId` 当前就是 document search 的最小工作区 / 项目边界，权限资源和底层 collection 都收敛到 `rag_${projectId}`。
- 文档级结果为空时，接口会稳定返回 `200 + results: [] + totalCandidates: 0`，不会将过滤后的空结果误报为命中。
- 底层 retriever 失败时，接口返回 `500`，并且不会误写成功检索指标。

### 3. 输出文档级证据

已完成。

依据：

- `server/rag/web-aigc-search-adapter.ts`
  已提供 `projectDocumentSearchResponse(...)`
- 该投影逻辑会先基于 `documentIds` 过滤结果，再按 `documentId` 聚合片段，并为每个文档输出：
  - `documentId`
  - `score`
  - `summary`
  - `fragments`
  - `highlights`
  - `metadata`
- `server/tests/rag-web-aigc-routes.test.ts`
  已有测试 `projects document_search results into document groups`
  测试明确验证：
  - 路由返回 200
  - 结果按文档聚合
  - 同一文档下存在多个 `fragments`
  - 文档级 `highlights` 已输出

结论：

“输出命中文档与片段，供下游消费”的核心闭环已经存在。

补充核查：

- `projectDocumentSearchResponse(...)` 会先应用 `documentIds` 过滤，再按 `documentId` 聚合文档级结果。
- 因此即使底层检索先返回候选片段，只要不在允许的 `documentIds` 范围内，最终返回也会是空数组。
- 已新增测试验证：当过滤后无文档可返回时，接口仍保持 `200`，并返回 `results: []`、`totalCandidates: 0`。

## 已完成但需限定边界的项

### 4. 验证权限与范围隔离

已完成，但应按“最小闭环”理解，不宜过度外推。

新增依据：

- `server/rag/retrieval/rag-retriever.ts` 使用 `projectId` 选择项目级 collection
- 同文件支持 `sourceTypes`、`sourceIds`、`agentId`、`codeLanguage` 过滤
- `server/rag/store/metadata-store.ts` 中也存在 `project_id`、`agent_id` 等元数据过滤逻辑
- `server/routes/rag.ts`
  - 在检测到全局 `PermissionCheckEngine` 时，会对 web-aigc document/fragment search 执行权限校验
  - 权限资源收敛为 `rag_${scope.projectId}`
  - 若缺少 `agentId` / `token`，在启用权限引擎时会直接返回 400
  - 若权限拒绝，会直接返回 403，不进入检索执行
- `server/tests/rag-web-aigc-routes.test.ts` 已验证：
  - 缺失 `query` 会报错
  - 缺失 `scope.projectId` 会报错
  - `fragment_search` 能按 `documentIds` 过滤结果
  - `document_search` 在 `documentIds` 过滤后为空时会返回空结果，并将检索指标记为无命中
  - 启用权限引擎后缺少 `agentId` / `token` 会返回 400
  - 权限拒绝会返回 403
  - 权限允许时仍可完成正常检索
  - 权限资源严格按 `rag_${projectId}` 绑定，不会脱离当前项目边界
  - retriever 失败时返回 500，且不误记成功检索指标

结论：

当前已经具备“权限 + 范围隔离”的最小闭环：

- 权限侧：请求会在路由入口绑定 `agentId + token + projectId` 做 `database:select` 校验
- 范围侧：检索仍受 `projectId` collection、`documentIds`、`sourceTypes` 等过滤约束
- 指标侧：文档级最终返回为空时按无命中统计，失败请求不记成功检索

但边界也很明确：

- 当前资源粒度是 `rag_${projectId}`，还不是文档级 ACL
- 当前依赖现有 capability token / permission matrix，并没有单独引入 web 端用户、租户、会话身份体系
- 当前也没有把 `documentIds` 与独立授权表做逐文档校验
- 当前错误返回仍是 `{ error: string }` 的最小结构，尚未补充更细粒度的失败分类

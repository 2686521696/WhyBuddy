# 条件节点现状核查

## 结论

基于当前 main 工作区代码，`web-aigc-node-condition` 的 4 个任务项里：

- 可以勾选 2 项
- 暂不能勾选 2 项

当前仓库已经有一条明确的 `web-aigc` 控制流快照承接链路，能够解析 `condition` 节点表达式、推导命中分支，并把结果投影到 graph instance 快照里；但还没有看到“条件判定事件”被明确写入 replay / audit 事件流的直接实现，也没有看到专门覆盖“异常表达式处理”的测试。

## 暂不能勾选项

### 3. 写入条件判定事件

暂不能勾选。

依据：

- `shared/web-aigc-observability.ts:43`
- `shared/web-aigc-observability.ts:53`
- `shared/web-aigc-observability.ts:70`
- `shared/web-aigc-observability.ts:105`
  观测目录里定义了 `node.started / node.completed / node.failed / edge.transitioned`
- 但本次核查范围内，没有看到 condition 判定结果被真正写成 replay / audit 事件的实现入口
- 当前看到最接近的是人类决策事件：
  `server/audit/audit-hooks.ts:229`
  `server/audit/audit-hooks.ts:336`
  `server/audit/audit-hooks.ts:349`
  它记录的是 `human.decision_submitted`，不是 condition 节点自动判定事件

结论：现在更像“已有观测事件 catalog + graph 快照投影”，还不能保守视为“条件判定事件已落地写入”。

### 4. 验证异常表达式处理

暂不能勾选。

依据：

- `server/core/web-aigc-controlflow.ts` 虽然有 `errors?: Record<string, string>` 和 `EXCEPTION` 状态映射，但当前表达式求值逻辑 `evaluateSimpleExpression(...)` 本身非常轻量，没有看到专门的异常分支抛错与兜底处理测试
- 本次检索未发现专门覆盖以下场景的测试：
  - 非法 / 畸形 expression
  - 不支持操作符
  - 表达式求值异常后写入 `errors`
  - condition 节点进入 `EXCEPTION`
- 现有相关测试主要覆盖正常分支命中：
  `server/tests/workflow-graph-projection.test.ts:220`
  `server/tests/workflow-graph-projection.test.ts:398`

结论：缺少直接的坏表达式 / 异常表达式测试证据，不能勾选。

## 现有能力落点

当前最明确的落点在 graph-instance / monitoring projection，而不是 runtime event stream：

- `server/core/web-aigc-controlflow.ts`
  `condition` 节点建模、表达式求值、branchHits 推导、blocked edge 计算
- `server/core/workflow-graph-projection.ts`
  把 controlflow 快照挂进统一 graph instance 投影
- `shared/workflow-graph.ts`
  提供节点运行快照与边状态结构
- `server/tests/workflow-graph-projection.test.ts`
  覆盖正常 condition 分支场景

## 还差什么

如果要让剩余 2 项也可勾，最少还需要补：

- 把 condition 判定结果明确写入 replay / audit 事件流
  例如落地 `edge.transitioned` / `node.completed` 的真实生产代码，而不只是 catalog 定义
- 增加异常表达式处理与测试
  例如非法表达式、空表达式边界、不支持操作符、求值失败时的 `errors` / `EXCEPTION` 输出

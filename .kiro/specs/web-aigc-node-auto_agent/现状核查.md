## 当前状态边界

虽然第 4 项已经可以勾选，但它的能力边界仍然需要说清楚：

- 当前 timeout 是执行器侧的 Promise race 软超时
  - 它会让调用方尽快返回错误
  - 但不等于底层 agent / internal_api 一定被真正取消
- 当前 retry 是固定次数的立即重试
  - 没有指数退避
  - 没有按错误类型差异化重试
- 当前 fallback 只支持一个显式后备目标
  - 还不是多级 fallback 链
- 当前恢复策略主要由 `metadata` 驱动
  - 还没有独立的强类型 request 字段与外部文档接口契约

这些限制不影响“任务已完成”的判断，但影响后续扩展路线。

## 后续可继续增强但不再阻塞当前勾选的方向

如果后面还想继续把这条链路做强，可以再补：

1. 把 `timeoutMs / retryCount / fallback` 升级成显式 request 字段，而不是只从 `metadata` 读取
2. 按错误类型区分可重试失败、不可重试失败、治理失败、超时失败
3. 为 retry 增加 backoff / jitter
4. 为 fallback 增加多级链路与策略选择
5. 为超时场景接入真正的取消能力

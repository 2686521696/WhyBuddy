# `confirm_judge` 现状核查

## 1. 结论摘要

基于当前仓库最新实现，`.kiro/specs/web-aigc-node-confirm_judge/tasks.md` 的 4 个任务项都已经有对应落地证据，可全部勾选。

原因是：

- 审计事件已经有明确实现入口、审计钩子和自动化测试覆盖。
- “批准 / 驳回”不再只是 HITL 决策元数据记录，当前 `WorkflowRuntimeEngine` 已内置 `confirm_judge` 适配器，可在恢复执行时基于 `branchKey` 命中 conditional edge，并驱动后续分支节点继续运行到终点。

## 2. 已有实现对应关系

### 2.1 `confirm_judge` 已进入共享节点与 HITL 模型

已有共享契约把 `confirm_judge` 视为 Web-AIGC HITL 节点之一：

- `shared/mission/contracts.ts`
  - `WEB_AIGC_HITL_NODE_TYPES` 已包含 `confirm_judge`
  - `WebAigcHitlSubmissionMetadata` / `DecisionHistoryEntry` 已支持 `nodeType`、`sessionId`、`interactionId`、`branchKey`

这说明 `confirm_judge` 至少在“节点类型建模”和“提交元数据承载”层面已经接入主干 HITL 决策体系。

### 2.2 已复用现有 mission decision 接口

现有任务决策链路已经成型，可直接承接 `confirm_judge`：

- `server/routes/tasks.ts`
  - 提供 `POST /api/tasks/:id/decision`
  - 内部调用 `submitMissionDecision(...)`
- `server/tasks/mission-decision.ts`
  - 负责校验选项、补充说明、`submittedBy`、metadata，并构造 `DecisionHistoryEntry`
- `server/tasks/mission-store.ts`
  - `resolveWaiting(...)` 会把当前 decision 归档进 `decisionHistory`，然后将任务从 `waiting` 恢复到 `running`
- `server/tests/hitl-decision.test.ts`
  - 已验证 `submittedBy`、metadata、decision history 追加等通用决策接口行为

这与 spec 中“复用现有任务决策接口”的方向是一致的。

### 2.3 已有 approve / reject 选项模型与人类决策 UI

现有系统已经具备“批准 / 驳回”这一类确认决策的选项与展示能力：

- `shared/mission/decision-templates.ts`
  - `execution-plan-approval` 模板内置 `approve / reject / modify`
  - `stage-gate` 模板内置 `proceed / hold / abort`
- `client/src/components/tasks/DecisionPanel.tsx`
  - 会优先识别并渲染 approve / reject 两类按钮
- `client/src/components/tasks/DecisionHistory.tsx`
  - 已按 `approve / reject / request-info / escalate / multi-choice` 展示决策历史

说明 `human decision` 的输入与展示面已经具备。

## 3. “写入确认审计事件”为何可以勾选

### 3.1 审计钩子已存在

`server/audit/audit-hooks.ts` 已对 `MissionRuntime.emitDecisionSubmitted(...)` 做 hook：

- 在 mission decision 提交成功后，记录审计事件
- 事件类型为 `AuditEventType.DECISION_MADE`
- metadata 中写入：
  - `eventKey: "human.decision_submitted"`
  - `workflowId`
  - `instanceId`
  - `missionId`
  - `decisionId`
  - `nodeId`
  - `nodeType`
  - `submittedBy`
  - `optionId`
  - `optionLabel`
  - `freeText`
  - `interactionId`
  - `branchKey`

这已经满足“确认行为必须记录操作者、时间与理由”的主体要求，也与 `audit hooks` 的现有实现对齐。

### 3.2 已有自动化测试覆盖

`server/tests/hitl-decision.test.ts` 中已有专门测试：

- `POST /api/tasks/:id/decision records audit decision_submitted entry`

测试会验证：

- 产生 `AuditEventType.DECISION_MADE`
- `eventKey === "human.decision_submitted"`
- actor 为提交人
- mission / node / session / decision / branchKey 等字段写入成功

因此，“写入确认审计事件”不只是设计意图，而是已有可运行证据与测试证明，建议勾选。

## 4. “验证批准与驳回分支”为何现在可以勾选

### 4.1 Runtime 已内置 `confirm_judge` 分支适配器

`server/core/workflow-runtime-engine.ts` 现在已经注册：

- `new HitlChoiceAdapter("confirm_judge", { branchFrom: "branchKey" })`

该适配器具备以下运行时行为：

- 首次执行时返回 `wait`，让实例进入 `WAITING_INPUT`
- 恢复执行时读取 `optionId / branchKey`
- 按 `branchKey` 在当前节点的 conditional edges 中查找同名 `label`
- 命中后将 `nextNodeId` 指向对应后继节点

这说明 `confirm_judge` 不只是“提交一个人工决策”，而是已经成为图运行时可以消费的 HITL 分支节点。

### 4.2 Runtime 恢复后会继续自动推进后续节点

`WorkflowRuntimeEngine.resume(...)` 现已具备以下逻辑：

- 如果等待节点恢复后实例重新回到 `EXECUTING`
- 且已经不再持有 checkpoint
- 则继续调用 `runToCheckpoint(...)`

这意味着 `confirm_judge` 节点提交决策后，不会停留在“分支已决定但未执行”的中间态，而会继续推进到批准或驳回路径上的后续节点，直到下一个 checkpoint 或终点。

### 4.3 已有针对批准/驳回分支的运行时测试

`server/tests/workflow-runtime-engine.test.ts` 已新增/补齐以下验证：

- `routes confirm_judge nodes by branchKey after resume`

测试覆盖了完整链路：

1. `confirm_judge` 节点先进入 `WAITING_INPUT`
2. 恢复时提交：
   - `optionId: "reject"`
   - `branchKey: "rejected"`
3. runtime 命中 `label: "rejected"` 的 conditional edge
4. 实例跳转到 `no-node`
5. 下游节点执行完成，实例状态变为 `EXECUTED`
6. `instance.variables` 中保留：
   - `selectedOptionId`
   - `branchKey`
7. `instance.output` 保留的是下游终点节点的业务结果，而不是分支元数据
8. `edgeTransitions` 中对应分支边状态更新为 `executed`

这已经足以证明“批准 / 驳回”分支不只是被记录，而是被 runtime 实际消费并跑通。

## 5. 当前仍值得继续补强的点

虽然 spec 4 项都可以勾选，但仍有几处值得后续继续增强：

### 5.1 需要补更贴近 mission 场景的集成测试

当前分支验证已经在通用 runtime 层打通，但还可以继续补一条更靠近任务/mission 的集成测试，验证：

1. 任务进入 waiting
2. 前端或 API 提交确认决策
3. runtime 继续推进批准或驳回分支
4. 同时保留 decision history 与 audit 记录

### 5.2 可以补 replay / projection 侧可视化验证

目前 runtime 已会记录 `branchKey` 与 edge transition，但如果后续要让 Office 或 replay 面板更直观展示“本次确认走了哪条边”，可以继续把这一层映射补到投影与前端可视化上。

## 6. 本次任务收口建议

当前建议将 `tasks.md` 的 4 项全部视为已完成，并以 runtime + 审计 + 测试三条证据链共同支撑：

- mission decision 入口承接人工确认输入
- audit hook 记录确认行为
- runtime adapter 基于 `branchKey` 驱动批准/驳回后续分支

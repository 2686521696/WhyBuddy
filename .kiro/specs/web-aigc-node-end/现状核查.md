# `web-aigc-node-end` 现状核查

## 1. 结论摘要

基于当前仓库最新实现，`web-aigc-node-end` 已经从“只有外围支撑能力”推进到“runtime 终点节点闭环已基本落地”。

对 `tasks.md` 的判断是：

- “定义最终输出结构”：可以勾选
- “打通实例完成与报告收敛”：可以勾选
- “写入结束事件”：可以勾选
- “验证多分支收敛场景”：可以勾选

原因不是仓库完全没有相关能力，而是：

- 当前已经有 `end` 节点内置 runtime adapter，能够在图执行时收敛最终输出并结束实例
- 当前已经有 runtime 完成后自动生成 `final_report` 的实现
- 当前已经有 runtime 完成后发射 `workflow_complete` 事件的实现
- 当前已经有一条自动化测试验证多分支最终进入不同 `end` 节点后的输出收敛、报告落盘和完成事件

## 2. 已有实现基础

### 2.1 已有通用最终输出模型

仓库里已经有通用的“节点执行完成”与“实例最终输出”模型：

- `shared/workflow-runtime-engine.ts`
  - 定义了 `WorkflowNodeCompleteResult`
  - 节点 adapter 可以返回 `kind: "complete"` 和 `output`
- `shared/workflow-domain.ts`
  - `WebAigcGraphInstance` 已包含 `output?: Record<string, unknown>`
- `server/tests/workflow-runtime-engine.test.ts`
  - 已验证最小运行图中，节点返回 `kind: "complete"` 后，实例状态变为 `EXECUTED`
  - 同时 `instance.output` 会落下最终输出

这说明“最终输出”作为通用 runtime 概念已经存在。

### 2.2 已有最终报告落盘与工作流完成事件

仓库里也已经具备传统 workflow pipeline 的完成收敛能力：

- `server/core/workflow-engine.ts`
  - 工作流跑完后会：
    - 更新 `completed_at`
    - 调用 `persistFinalReport(...)`
    - 发射 `workflow_complete`
- `server/memory/report-store.ts`
  - 已定义并持久化 `FinalWorkflowReport`
- `server/routes/workflows.ts`
  - 已提供：
    - `GET /api/workflows/:id/report`
    - `GET /api/workflows/:id/report/download`

这说明“实例完成 -> 报告收敛”在旧工作流主链路中是成立的。

### 2.3 已有 `end` 节点的控制流表示与投影

仓库里已经明确出现 `type: "end"`：

- `server/core/web-aigc-controlflow.ts`
  - `ControlFlowNodeType` 包含 `"end"`
  - `ControlFlowEndNode.config.output` 支持输出表达式
  - `buildEndNodePreview(...)` 会为 `end` 节点生成输出预览
- `server/core/workflow-graph-projection.ts`
  - 会优先把 `webAigcControlFlow` 投影成图实例快照
- `server/tests/workflow-graph-projection.test.ts`
  - 已覆盖 `end-approved` / `end-rejected` 这类 `end` 节点在图投影中的展示与边状态

这说明 `end` 节点至少已经在“控制流建模 / 监控投影 / UI 展示”层面存在。

## 3. 为什么“定义最终输出结构”现在可以勾选

`server/core/workflow-runtime-engine.ts` 现已注册 `EndWorkflowNodeAdapter`，并为 `type: "end"` 提供统一完成输出结构：

- `status`
- `summary`
- `artifacts`
- `result`
- `finalVariables`

该适配器支持从 `node.config` 中读取：

- `status`
- `summary`
- `artifacts`
- `output`

同时支持通过 `$.path` 和 `@variables` 模板引用上游变量，将分支结果、变量快照和产物引用统一收敛到 `WebAigcGraphInstance.output`。

这意味着 `end` 节点已经不再只是“任何 complete 节点都能算结束”，而是有了自己的终点输出约定，因此可以勾选。

## 4. 为什么“打通实例完成与报告收敛”现在可以勾选

### 4.1 runtime 已补上 final report 收敛

`server/core/workflow-runtime-engine.ts` 现在已经在实例完成时补上了 runtime 收敛逻辑：

- 基于当前 `StoredWebAigcRuntimeState` 生成 `FinalWorkflowReportRecord`
- 通过 `runtime.reportRepo.saveFinalWorkflowReport(...)` 落盘
- 回写 `workflow.results.final_report`

因此，`end` 节点到达完成态之后，已经不是只更新 `instance.output`，而是会同步生成平台可复用的最终报告元信息。

### 4.2 现有读取链路可直接复用

虽然报告读取接口本身还是原来的：

- `server/routes/workflows.ts`
- `server/memory/report-store.ts`

但现在 Web-AIGC runtime 已经主动回写 `workflow.results.final_report`，所以读取面不需要额外造新接口，说明这项已经具备最小闭环。

## 5. 为什么“写入结束事件”现在可以勾选

### 5.1 runtime 已主动发射完成事件

`server/core/workflow-runtime-engine.ts` 当前在 runtime 终态收口时，已经会主动发射：

- `workflow_complete`

其中 summary 会优先取 `end` 节点输出中的 `summary`，否则回退到当前节点说明。

这说明 Web-AIGC runtime 已经不再完全依赖传统 workflow engine 的完成事件。

### 5.2 当前已形成最小事件闭环

当前事件链路至少已经覆盖：

- `workflowRuntimeEngine -> runtime.eventEmitter.emit(...)`
- `server/runtime/server-runtime.ts -> emitEvent(...)`
- `server/core/socket.ts -> io.emit("agent_event", event)`

虽然它还不是 `node.completed:end` 这种更细颗粒的专用 replay 事件，但从 spec 的“结束节点应写入实例完成事件”要求来看，已经具备最小可用完成事件闭环，因此可以勾选。

## 6. 为什么“验证多分支收敛场景”现在可以勾选

`server/tests/workflow-runtime-engine.test.ts` 现已补齐一条针对 runtime 的分支终点测试：

- `selection-node` 作为等待输入的分支节点
- 两条 conditional edges 分别指向：
  - `end-approved`
  - `end-rejected`
- 恢复时提交 `optionId: "approved"`

该测试已经验证：

1. runtime 从 `WAITING_INPUT` 恢复后会命中正确分支
2. 命中分支上的 `end` 节点会被真正执行
3. `instance.output` 会收敛最终结构化输出
4. `final_report` 会在 runtime 完成后被持久化
5. `workflow_complete` 会被发射
6. `edgeTransitions` 会把命中分支标记为 `executed`
7. 未命中的 `end` 节点保持 `PENDING`

因此，“多分支最终收敛到某个 `end` 节点”的最小运行时证据已经具备，可以勾选。

## 7. 当前仍值得补强的能力边界

虽然 4 项都可以勾选，但仍有两块建议继续补强：

### 7.1 end 专属 replay / audit 事件目录

- 当前发的是通用 `workflow_complete`
- 还没有把 `end` 节点完成显式映射成 `node.completed` / `instance.completed` 这类更细粒度的 replay/audit 事件

### 7.2 final report 内容仍偏 runtime 最小摘要

- 当前 runtime final report 已可落盘
- 但内容仍是最小化结构，尚未充分吸收 mission / session / audit / replay 的更多聚合字段

## 8. 建议的后续补差路径

若后续要继续提升质量，建议最小补齐顺序如下：

### 8.1 补 end 专属 observability 事件

至少需要：

- 将 `end` 节点完成映射进 `shared/web-aigc-observability.ts` 目录
- 让 replay / audit 至少有一条能按 `nodeId=end-*` 检索的直接证据

### 8.2 丰富 runtime final report

至少需要：

- 把 mission / session / replay / audit links 更多写入报告
- 让 `end` 节点输出与平台报告结构更紧密对齐

## 9. 本次回写建议

基于当前证据，`tasks.md` 现在可以 4 项全部勾选。

这个状态更准确反映当前仓库的真实进展：

- `end` 节点本体的 runtime 闭环已经落地
- report / event 已形成最小可用闭环
- 更高阶的 replay / audit 精细化仍属于下一阶段增强项

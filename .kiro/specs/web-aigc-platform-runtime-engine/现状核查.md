# Web-AIGC 平台运行时引擎现状核查

## 核查结论

截至 2026-04-22，`web-aigc-platform-runtime-engine` 这组任务更准确的状态应为“4 项成立，1 项部分完成但未完全闭环”。

- 已成立：任务 1、任务 2、任务 3
- 已成立：任务 5
- 尚未完全闭环：任务 4
- 本轮新增并已落地的关键能力：`selection`、`confirm_judge`、`end`、显式 `terminate / retry / escalate`、节点级自动重试/自动升级

本次核查已经基于 `server/core/workflow-runtime-engine.ts`、`server/core/web-aigc-runtime-observability.ts`、`server/routes/workflows.ts`、`server/tests/workflow-runtime-engine.test.ts`、`server/tests/workflows-routes.test.ts`、`server/tests/web-aigc-runtime-observability.test.ts` 重新校准结论，不再沿用上一轮旧判断。

## 当前已成立的能力

### 1. 节点执行适配器接口已成立

运行时已经存在统一的节点适配器契约，支持：

- `execute(context)` 执行节点
- `resume(context)` 恢复等待中的节点
- 返回 `advance / wait / complete / error` 四类标准结果

这说明“节点如何被引擎调度”已经有稳定抽象，不再依赖单一线性流程写死逻辑。

### 2. 图跳转与状态推进已成立

运行时已经具备以下基础图执行能力：

- 依据 `edgeSchemas` 决定默认下一个节点
- 支持 `conditional` 边按标签命中分支
- 在实例上记录 `edgeTransitions`
- 为每个节点维护 `PENDING / EXECUTING / WAITING_INPUT / EXECUTED / EXCEPTION` 等运行态

从现状看，平台已经不是“只会从左到右串行推进”的状态，而是具备基础图运行语义。

### 3. 等待输入与恢复执行机制已成立

运行时已经支持：

- 节点返回 `wait` 后写入 `checkpoint`
- 在实例中保留 `waitingFor / inputSchema / payload`
- 外部通过 `resume(payload)` 恢复
- 恢复后若实例回到 `EXECUTING` 且没有新的 `checkpoint`，引擎会自动继续 `runToCheckpoint()`，直到下一个 checkpoint 或终点

这说明 HITL 并不是停留在模型层约定，而是已经进入实际运行时链路。

## 本轮重点校准的 3 个运行时能力

### `selection` 已落地为内置分支节点

当前运行时已内置 `selection` 适配器，具备以下行为：

- 首次执行时进入 `WAITING_INPUT`
- 通过节点 `config.options` 生成可选项
- 恢复时读取 `optionId / optionIds / selectedOptionIds`
- 按 `conditional edge.label` 命中分支
- 将 `selectedOptionId / selectedOptionIds / branchKey / selectedLabel` 回写到实例变量

这意味着 `selection` 已经不只是节点定义存在，而是可以真实驱动分支路由。

### `confirm_judge` 已落地为内置人工决策节点

当前运行时已内置 `confirm_judge` 适配器，具备以下行为：

- 首次执行进入等待态，形成可恢复 checkpoint
- 恢复时可读取 `optionId` 与 `branchKey`
- 以 `branchKey` 为优先分支键命中 `approved / rejected` 等条件边
- 决策结果会沉淀到实例变量，供后续节点继续消费

这说明批准/驳回类节点已经进入真实执行闭环，而不是仅停留在 spec 语义。

### `end` 已落地为内置终态收敛节点

当前运行时已内置 `end` 适配器，已经可以在流程末端做结构化收敛：

- 支持从节点配置读取 `status / summary / artifacts / output`
- 支持使用 `$.path` 从实例变量取值
- 支持使用 `@variables` 引用完整变量快照
- 最终输出会收敛为结构化结果，包含：
  - `status`
  - `summary`
  - `artifacts`
  - `result`
  - `finalVariables`

这代表运行时已经从“把节点跑完”进一步升级到“在终点做统一输出收敛”。

同时需要明确两个输出语义边界：

- 节点执行结果中的 `result.output` 会并入 `instance.variables`，作为后续节点共享变量继续流转
- `instance.output` 代表当前实例的终态产出；在 `end` 节点完成时，它以终点节点收敛结果为主，不等于“自动合并全部 variables 作为最终输出”

这一区分可以避免把“过程变量”误当成“最终交付结果”。

## 路由控制面已经形成最小可用入口

当前 `server/routes/workflows.ts` 已对运行时暴露以下接口，并且 `server/tests/workflows-routes.test.ts` 已覆盖最小行为验证：

- `GET /api/workflows/:id/runtime-definition`
- `GET /api/workflows/:id/runtime-state`
- `POST /api/workflows/:id/runtime/run`
- `POST /api/workflows/:id/runtime/resume`
- `POST /api/workflows/:id/runtime/terminate`
- `POST /api/workflows/:id/runtime/retry`
- `POST /api/workflows/:id/runtime/escalate`

这意味着当前运行时不只是“引擎类存在”，而是已经具备可被平台侧显式调用的主仓控制入口。

## 为什么任务 4 暂时不能继续勾选

任务 4 原表述为“接入终止、重试、失败升级策略”。

当前真实情况是：

- 已有异常收敛：节点返回 `error` 时，实例会进入 `EXCEPTION`
- 已有终止态模型：领域状态中存在 `FORCE_TERMINATED`
- 已有 `retryable` 类型字段：适配器错误结果可声明是否可重试

但这轮之后，真实情况已经变成“两段式结论”：

- 已经完成的部分：
  - 运行时引擎已有显式 `terminate()`、`retry()`、`escalate()` 入口
  - `server/routes/workflows.ts` 已暴露三类 runtime 控制 API
  - `retryable` 已被 `retry()` 入口消费为最小重试准入条件
  - `escalate()` 已把失败实例显式转成 `WAITING_INPUT + human escalation review` 检查点
  - `terminate()`、`retry()`、`escalate()` 都会发出结构化 runtime 控制事件
  - 节点配置现已可通过 `retryBudget / retryDelayMs / autoEscalateOnFailure / escalateOnRetryExhausted` 驱动最小自动策略
  - 引擎已支持“自动重试成功”与“重试耗尽后自动升级到人工 review”两种基础自动策略路径

- 仍未完成的部分：
  - 没有统一的后台重试队列、指数退避策略中心或全局预算治理
  - `retryable` 虽已进入节点级自动策略，但还没有演进成平台统一调度策略
  - 失败升级虽已支持节点级自动升级，但还不是跨节点、跨阶段的自动治理编排
  - 还没有覆盖真正的端到端治理联动，例如“失败后自动升级审批，再恢复执行”

因此，这一项现在应该理解为“最小显式控制面 + 最小自动策略层已成立，但完整策略闭环仍未完成”，所以仍不应整体标记为已完成。

## 为什么任务 5 现在可以勾选

任务 5 原表述为“将事件接入 replay / telemetry / audit”。这项在当前最小闭环层面已经成立，但 `telemetry` 的现实形态需要说得更精确。

当前真实情况是：

- 运行时状态会持久化到 `workflow.results.webAigcRuntime`
- 实例链接中已经保留 `replayId / auditId / missionId / sessionId`
- 引擎内部现在会统一发射：
  - `node.started`
  - `node.completed`
  - `node.waiting_input`
  - `node.failed`
  - `edge.transitioned`
- 这些事件会通过既有 `runtime.eventEmitter -> emitEvent -> socket(agent_event)` 通道进入平台统一事件面
- `server/core/web-aigc-runtime-observability.ts` 已把 runtime 事件镜像到 replay / audit
- 新增控制事件：
  - `instance.terminated`
  - `instance.retry_requested`
  - `instance.escalated`
  也已进入 replay / audit 映射与测试

更准确地说，当前是“统一事件出口与 replay / audit 镜像都已成立”；其中 `telemetry` 更接近“进入平台公共事件面”，不是独立遥测后端；`lineage` 关联模型虽然已在共享层定义，但 runtime 事件当前并未直接写入 lineage。

因此，这一项现在可以勾选，但需要把口径限定为：

- 运行时节点事件已经统一发射
- 事件已经进入平台公共事件通道
- replay / audit 的最小镜像已经成立
- 更细粒度的 telemetry backend 与 runtime -> lineage 直写仍属于下一阶段增强，不影响本项最小闭环成立

## 对当前 spec 的准确理解

现在的运行时引擎已经完成了“图执行 + HITL 分支 + 终态收敛”的核心底座，这也是为什么：

- `selection` 可以真实驱动条件边
- `confirm_judge` 可以真实驱动批准/驳回
- `end` 可以在分支末端产出结构化终态结果

但它还没有完全进化到“带统一治理策略中心、独立遥测后端、runtime 直连 lineage 的完整版运行时内核”。

因此，更准确的判断是：

- 运行时主干能力已经成立
- 显式控制面已经从 0 推进到了最小可用
- 节点级自动策略已经从 0 推进到了最小可配
- 自动治理策略闭环仍在后续批次中

## 建议后续收口方向

后续若要把本 spec 重新推进到“全部勾选”，建议按以下顺序补齐：

1. 把当前节点级自动策略继续升级为平台统一可配置策略层
2. 让 `retryable` 参与更完整的自动调度决策，而不只服务于单节点策略
3. 把 runtime 控制事件继续接入更完整的 lineage / 治理链路
4. 再补一组端到端验证，证明 runtime 控制、审计、回放、人工恢复已经贯通

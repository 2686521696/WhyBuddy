# web-aigc-node-selection 现状核查

## 结论

截至 2026-04-22，`web-aigc-node-selection` 在当前主仓中已经形成双层闭环：

- 一层是既有的 `MissionDecision + waiting/resume + tasks route + audit hook` 通用人工输入底座
- 一层是主仓现在已经内置的 `selection` runtime adapter，用于图运行时等待、恢复、分支跳转与事件发射

因此，`tasks.md` 的 4 个任务项目前都可以保留为已完成状态。

和之前相比，最需要纠正的一点是：

- 现在不能再把 `selection` 描述成“只有任务决策复用，没有独立 selection 运行时”
- 主仓已经在 `server/core/workflow-runtime-engine.ts` 中内置了 `HitlChoiceAdapter("selection")`

## 分任务判断

### 1. 定义选项结构与校验规则

结论：可勾选。

依据：

- `shared/mission/contracts.ts`
  - 已定义 `MissionDecisionOption`，包含：
    - `id`
    - `label`
    - `description`
    - `action`
    - `severity`
    - `requiresComment`
  - 已定义：
    - `MissionDecision`
    - `MissionDecisionSubmission`
    - `MissionDecisionResolved`
  - 已将 `"selection"` 纳入 `WEB_AIGC_HITL_NODE_TYPES`
  - `WebAigcHitlFieldDefinition.type` 已支持 `"selection"`
- `server/tasks/mission-decision.ts`
  - 已对以下规则做服务端校验：
    - `optionId` 必须存在于候选项中
    - `requiresComment=true` 时必须提交 `freeText`
    - 不允许空选择
    - `allowFreeText !== true` 时不允许非法自由文本
- `server/core/workflow-runtime-engine.ts`
  - `HitlChoiceAdapter("selection")` 在图运行时已补上 selection 自身的最小校验：
    - wait 时输出 `inputSchema`
    - resume 时支持 `optionId / optionIds / selectedOptionIds / branchKey`
    - 未提供选择时返回 `Missing required selection payload`
    - 候选项不匹配时返回 `Selected option is not defined...`
- `server/tests/hitl-decision.test.ts`
  - 已验证 `requiresComment` 等决策校验
- `server/tests/workflow-runtime-engine.test.ts`
  - 已验证 selection runtime 的等待、恢复和分支推进

说明：

- 这里的“选项结构与校验规则”已经同时被 Mission decision 契约和 selection runtime adapter 覆盖，不需要节点另起一套。

### 2. 复用任务输入接口

结论：可勾选。

依据：

- `server/routes/tasks.ts`
  - `POST /api/tasks/:id/decision` 已作为统一人工输入入口处理选择提交
- `server/tasks/mission-runtime.ts`
  - 已提供 `waitOnMission(...)` 与 `resumeMissionFromDecision(...)`
- `server/tasks/mission-decision.ts`
  - 提交后会把 resolved 结果写回 Mission decision history，并驱动任务恢复
- `server/core/workflow-runtime-engine.ts`
  - selection 节点现在也能直接通过 runtime checkpoint + `resume(...)` 恢复执行
- `client/src/components/tasks/DecisionPanel.tsx`
  - 已接入等待态任务输入面板
- `client/src/lib/mission-client.ts`
  - 已接入决策提交 API
- `client/src/lib/tasks-store.ts`
  - 已消费 decisionSubmitted 结果
- `server/tests/mission-routes.test.ts`
  - 已验证 `POST /api/tasks/:id/decision` 提交后，任务从 waiting 恢复为 running
- `server/tests/workflow-runtime-engine.test.ts`
  - 已验证 selection 节点进入等待、提交后恢复并继续走分支

说明：

- spec 设计里写的是 Cube 承接 `server/routes/tasks.ts`，这点仍然成立；
- 但当前事实已经比原设计更进一步，selection 还具备了 runtime 内置执行能力。

### 3. 写入选择结果事件

结论：可勾选。

依据：

- `shared/web-aigc-observability.ts`
  - 已定义：
    - `node.waiting_input`
    - `human.decision_submitted`
- `server/core/workflow-runtime-engine.ts`
  - selection 节点 wait 时会发出 `node.waiting_input`
- `server/routes/tasks.ts`
  - `POST /api/tasks/:id/decision` 成功后会调用 `runtime.emitDecisionSubmitted(...)`
- `server/tasks/mission-runtime.ts`
  - 已实现 `emitDecisionSubmitted(...)` Socket 广播
- `server/audit/audit-hooks.ts`
  - 已将决策提交映射为审计链中的 `human.decision_submitted`
- `server/tests/hitl-decision.test.ts`
  - 已验证审计链里存在 `human.decision_submitted`
  - 测试中明确覆盖：
    - `nodeType: "selection"`
    - `nodeId`
    - `optionId`
    - `optionLabel`
    - `branchKey`
- `server/tests/workflow-runtime-engine.test.ts`

说明：

- 当前“选择结果事件”既有任务决策提交后的审计/回放侧事件，也有图运行时等待态事件；
- 按当前 spec 粒度，这已经足够证明闭环存在。

### 4. 验证单选与多选场景

结论：可勾选。

依据：

- `server/core/workflow-runtime-engine.ts`
  - 单选场景支持 `optionId`
  - 多选场景支持 `optionIds / selectedOptionIds`
  - `multiple / selectionMode=multiple|multi|multi-select|multi-choice` 会切换为数组型输入 schema
- `server/tests/workflow-runtime-engine.test.ts`
  - 已验证 selection 节点按单选结果走不同分支
  - 已验证结果写回：
    - `selectedOptionId`
    - `selectedOptionIds`
    - `branchKey`
  - 已验证选择后继续进入 end 节点并完成结构化输出
- `server/tests/hitl-decision.test.ts`
  - 已验证 `type: "multi-choice"` 的任务决策路径和 metadata/history 持久化
- `client/src/components/tasks/DecisionPanel.tsx`
  - 已有 `multi-choice` 展示分支

说明：

- 图运行时侧已经明确支持数组型输入；
- 任务决策侧的“多选”语义仍偏向 `multi-choice` 决策类型，而不是完整的数组型多选提交；
- 但就当前 spec 文义和已有实现来看，单选与多选两个场景都已有可验证证据，可以继续勾选。

## 当前能力边界

虽然 4 个任务项都可以勾选，但 selection 仍然有几个值得后补的边界：

1. 任务决策接口和 graph runtime 的“多选”语义还没有完全统一
   - runtime 已支持 `optionIds / selectedOptionIds`
   - task route 侧主要仍是单个 `optionId` + `multi-choice` 决策类型

2. 缺少 selection 节点更强的端到端 replay 专项测试
   - 当前已有 runtime event 和 audit event 证据
   - 但 replay 视图层的 selection 节点专项断言仍可继续补强

3. 缺少更显式的 selection 专用前端动态多选输入面板
   - 现有 `DecisionPanel` 已支持 `multi-choice`
   - 但还不是图节点驱动的专用 selection UI 配置面

## 对 tasks.md 的判断

本次核查后，`tasks.md` 的 4 项全部保持已完成是准确的。

需要同步修正的是叙述口径：

- 应从“主要依赖 Mission decision 复用”升级为“既复用任务输入接口，也已有内置 selection runtime adapter”

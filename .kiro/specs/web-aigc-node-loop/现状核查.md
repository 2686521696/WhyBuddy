# web-aigc-node-loop 现状核查

## 结论

当前主仓已经具备一部分与 `loop` 节点相关的基础能力，包括：

- 域模型里预留了 `loop` 边种类；
- observability 目录里定义了 `edge.loop_iterated` 事件；
- 轻量 runtime 已支持 `maxSteps`、`WAITING_INPUT`、`resume`；
- monitoring 路由已支持人工 `terminate`。

但这些能力还没有收敛成一个真正的 `loop` 节点闭环。当前能直接确认的控制流实现仍以 `start / variable_assignment / condition / end` 为主，没有看到 `loop` 节点类型、循环配置、迭代执行、迭代级 replay 写入的生产实现。

因此 `tasks.md` 里的 4 个任务项当前都不建议勾选。

## 分任务判断

### 1. 定义循环类型与配置

结论：暂不勾选。

已有证据：

- `shared/workflow-domain.ts`
  - `WEB_AIGC_EDGE_KINDS` 已包含 `"loop"`。
- `shared/workflow-domain.ts`
  - `WebAigcEdgeKind` 和相关 edge record 已允许 `loop` 作为边种类。

直接缺口：

- `server/core/web-aigc-controlflow.ts`
  - 当前 `ControlFlowNodeType` 只包含 `start / variable_assignment / condition / end`。
- 没有看到 `loop` 节点专用配置，例如：
  - 固定次数循环
  - 条件循环
  - 列表迭代
  - 循环变量 / 迭代上下文
- 没有看到 shared contract 定义 `loop` 节点的 config schema。

判断依据：

- 现有代码更像“loop 边类型预埋”，不是“loop 节点定义已完成”。

### 2. 增加次数与耗时保护

结论：暂不勾选。

已有证据：

- `server/core/workflow-runtime-engine.ts`
  - `runToCheckpoint(...)` 有 `maxSteps`，说明轻量 runtime 已有基础步数保护。
- `server/core/a2a-client.ts`
  - 存在 `terminateTimedOutSessions()`，说明仓库整体已有 timeout 治理惯例。

直接缺口：

- 没有看到 `loop` 节点自己的：
  - `maxIterations`
  - `maxDurationMs`
  - 每轮耗时累计
  - 超限后的 loop 专属错误或停止策略
- 当前 `maxSteps` 是整个轻量 runtime 的通用限制，不等于 loop 节点的次数与耗时保护。

判断依据：

- spec 要求的是 `loop` 节点级安全边界，目前只能看到 runtime 通用步数限制，不能视为完成。

### 3. 写入迭代级回放事件

结论：暂不勾选。

已有证据：

- `shared/web-aigc-observability.ts`
  - 已定义 `edge.loop_iterated` 事件。
  - required fields 包含 `loopKey`、`iterationIndex`。

直接缺口：

- 全仓检索只看到事件目录定义，没有看到生产代码实际写入 `edge.loop_iterated`。
- `shared/workflow-graph.ts`
  - 当前 `GraphEdgeTransitionSnapshot.kind` 仅支持 `"parent_child" | "control_flow"`。
- `shared/aigc-monitoring.ts`
  - 当前监控 edge detail 也仅支持 `"parent_child" | "control_flow"`。
- 现有 graph projection / replay 相关测试只覆盖了 condition 控制流，没有 loop 迭代回放样例。

判断依据：

- 事件目录已经预留，但 replay 数据结构和落库/投影链路还没有形成 loop 迭代闭环。

### 4. 接入人工终止与超时处理

结论：暂不勾选。

已有证据：

- `server/routes/aigc-monitoring.ts`
  - 已支持 `/instances/:instanceId/terminate`。
- `server/tests/aigc-monitoring-routes.test.ts`
  - 已验证 monitoring terminate 会复用 mission terminate 流程。
- `server/core/workflow-runtime-engine.ts`
  - 已支持 `WAITING_INPUT`、checkpoint、`resume(...)`。
- `server/tests/workflow-runtime-engine.test.ts`
  - 已验证 checkpoint 与 resume。

直接缺口：

- 这些能力目前是“实例级 terminate / 通用等待恢复”，不是 `loop` 节点专用中断治理。
- 没有看到：
  - 正在执行的 loop 迭代被人工终止
  - loop 超时后自动停止
  - 终止后回放/状态中记录 loop 中断点

判断依据：

- spec 关注的是 `loop` 节点与取消链路打通；当前只能证明“运行时有 terminate / wait / resume”，还不能证明 loop 已接入。

## 最接近闭环、可优先补的点

1. 先在控制流定义里增加 `loop` 节点类型与 config schema
   - 至少覆盖固定次数循环；
   - 再逐步扩展条件循环、列表迭代。

2. 在轻量 runtime 为 `loop` 节点补最小执行闭环
   - 维护 `iterationIndex`；
   - 支持 `maxIterations`；
   - 把每轮输入/输出摘要写入 runtime state。

3. 让 replay / projection 真正消费迭代事件
   - 至少能输出 `edge.loop_iterated`；
   - 让 monitoring / graph snapshot 能看到循环迭代轨迹。

4. 再把 terminate / timeout 精确挂到 loop 执行路径
   - 支持“循环中断于第几轮”的状态表达。

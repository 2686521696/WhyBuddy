# 消息通知节点落地现状

## 一、当前已落地的真实能力

仓库里已经存在一条以飞书为主的通知能力链路，可以作为 `message_notification` 节点的第一阶段承接面：

- `server/routes/feishu.ts`
  - 提供统一飞书路由入口。
- `server/feishu/runtime.ts`
  - 负责装配 task store、bridge、delivery、dispatcher、去重存储。
- `server/feishu/bridge.ts`
  - 负责把任务状态变化转换成可发送的通知消息。
- `server/feishu/delivery.ts`
  - 负责飞书 API 的真实发送与更新。
- `server/feishu/relay.ts`
  - 提供 relay 请求与 relay 事件更新入口。
- `server/feishu/ingress.ts`
  - 处理 webhook 入站、去重、卡片回调和任务启动。

这条链路已经能完成：

- 复杂请求进入后立即 ACK
- 任务进度更新通知
- 等待确认通知
- 完成/失败终态通知
- 飞书卡片按钮回调
- 线程内回复与消息更新

## 二、与 4 个 task 的对应判断

### 1. 定义通知输入输出结构

这一项已经有明确代码证据，且结构较完整：

- `FeishuTaskTarget` 定义目标会话、线程、消息引用
- `FeishuOutboundMessage` 定义通知输出结构
- `FeishuDeliveryReceipt` 定义发送回执
- `FeishuTaskRequest` / `FeishuTaskRequestAck` 定义请求进入与 ACK 输出
- `FeishuTaskRecord` / `FeishuTaskEvent` 定义通知驱动所依赖的任务态

因此可以认定“首条通道的通知输入输出结构”已经落地。

### 2. 设计消息模板能力

这一项在飞书场景下也已有明确证据：

- 文本模板函数已覆盖 ACK、进度、等待、完成、失败、决策已处理等消息类型
- 卡片模板支持按状态切换颜色
- 决策卡片支持按钮、风险等级、审批/升级样式
- 支持 `text / card / card-live` 三种展示模式
- 单测已验证卡片颜色、按钮样式、决策渲染、最终摘要

但当前模板能力仍偏“代码内置模板”，还不是一个独立的模板中心。

### 3. 增加发送审计记录

这一项现在可以勾选。

虽然本轮没有去改平台级 `server/audit/*` 主链，但已经在允许范围内把飞书通知链自身的发送审计补齐了：

- `server/feishu/bridge.ts`
  - 新增 `FeishuNotificationAuditEvent`
  - 在 `sendAndTrack()` 与 `sendFinalSummaryIfNeeded()` 中统一产出通知发送审计事件
- `server/feishu/delivery.ts`
  - 把真实发送/更新中的响应、限流、网络异常与重试过程折叠成 telemetry
  - 再由 bridge 统一写成发送审计事件
- `server/tests/feishu-bridge.test.ts`
  - 已验证发送成功时会记录统一审计事件
  - 已验证发送失败时会把 `retryCount / attempts / statusCode / error` 写入审计事件
- `server/tests/feishu-routes.test.ts`
  - 已验证 webhook 入口触发 ACK 通知时也会落发送审计事件

当前已经能稳定记录的字段包括：

- `auditId / traceId`
- `channelType / recipientScope / targetId`
- `templateId / messageDigest`
- `approvalStatus / resultStatus`
- `attemptCount / retryCount / attempts`
- `statusCode / error`
- `observability.progress / taskStatus / stage / presentation`

因此，更准确的判断已经不是“发送审计未落地”，而是“飞书通知链内的发送审计已落地，后续再决定是否镜像进平台统一 audit collector”。

### 4. 验证失败重试策略

这一项现在可以勾选。

当前已有实现：

- `FeishuApiDelivery.fetchWithRetry()`
- 对 `429 / 5xx` 的重试判断
- `Retry-After` 解析
- 指数退避与最大等待时间配置

本轮已经补了两层验证：

- `server/feishu/delivery.ts` 现在会把每次尝试的 `statusCode / retryable / delayMs / error` 汇总成 telemetry
- `server/tests/feishu-bridge.test.ts` 已验证这些 telemetry 会进入通知发送审计事件
- `server/tests/feishu-delivery.test.ts` 已新增 `FeishuApiDelivery` 的独立专项测试矩阵，覆盖：
  - `429 + Retry-After` 后重试成功
  - `network-error` 连续失败后重试耗尽
  - `400` 非重试型错误立即失败
  - token 获取阶段 `503` 后重试成功

这意味着当前已经不只是“实现了重试逻辑”，而是已经有了节点级专项验证证据。

仍然存在的边界是：

- 还没有覆盖“重试彻底耗尽后，外围任务编排如何降级、补偿或告警”的更高层端到端行为

但那已经超出 `message_notification` 节点自身“发送重试策略是否成立”的最小验证范围。

因此当前更准确的说法是：

- “重试逻辑已实现”
- “失败重试过程已能进入通知发送审计/观测链”
- “失败重试策略已完成节点级独立专项验证”

## 三、当前缺口

如果后续要把 `message_notification` 从“飞书通知薄切片”推进成更标准的 Web-AIGC 节点，还需要补下面几类能力：

1. 统一节点协议
   - 从飞书专用结构提升为通用 `channelType / recipients / templateId / variables / deliveryMode`。
2. 审计治理
   - 已在飞书通知链内补齐 `auditId / traceId / approvalStatus / resultStatus / retryCount` 等字段。
   - 后续可再评估是否需要镜像进平台统一 audit collector。
3. 多通道抽象
   - 目前证据几乎都集中在飞书，没有看到邮件、站内信、Webhook 等统一适配层。
4. 重试验证
   - 节点级重试验证已补齐。
   - 后续若进入平台联调阶段，可再补“任务编排级降级/补偿”端到端验证。

## 四、结论

`web-aigc-node-message_notification` 当前最准确的状态是：

- 已有“飞书通知”这一条可复用薄切片
- 已有通知结构与模板化渲染能力
- 已具备飞书通知链内的发送审计闭环
- 已完成失败重试策略的独立专项验证

因此这个 spec 当前更准确的推进状态是：

- 飞书通道最小闭环已存在
- 发送审计已补线完成
- 失败重试专项验证已补齐

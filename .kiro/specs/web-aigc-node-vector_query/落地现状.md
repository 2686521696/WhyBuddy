# 向量查询节点落地现状

## 复核说明

本文件已按 2026-04-22 当前仓库代码与共享契约重新核查。

本轮结论是：

- `tasks.md` 当前勾选状态仍然准确，无需调整
- 前 3 项可以继续保持已完成
- “写入查询监控事件”仍未真正接入查询链路，继续保持未完成
- 当前实现仍以文本 `query` 触发向量化检索，不支持外部直接传入原始向量数组

因此，本次仅补充复核说明，不改动 `tasks.md` 勾选。

## 一、总体判断

`web-aigc-node-vector_query` 在当前仓库里并不是一套独立的新检索引擎，而是对现有 RAG 检索能力的 Web-AIGC 兼容适配。

从现状看，这条 spec 已经完成了较大一部分基础闭环：

- 已有独立的 Web-AIGC 检索输入输出契约
- 已接到现有 RAG route + adapter + retriever 链路
- 已能返回分值、元数据、摘要、高亮等结果字段

但“查询监控事件写入”这一步还没有真正接上，因此目前更准确的结论是：

- `vector_query` 的检索接口与结果投影已落地
- 监控 / 事件化收口仍未完成

## 二、已落地能力

### 1. Web-AIGC 检索接口契约已存在

`shared/rag/web-aigc-search.ts` 已定义这条节点可直接复用的输入输出模型：

- `WebAigcSearchRequest`
- `WebAigcSearchScope`
- `WebAigcSearchOptions`
- `WebAigcDocumentSearchResponse`
- `WebAigcFragmentSearchResponse`
- `WebAigcSearchErrorResponse`

同时还定义了两个兼容路由常量：

- `POST /api/rag/web-aigc/document-search`
- `POST /api/rag/web-aigc/fragment-search`

这说明 `vector_query` 对外接口已经不是口头约定，而是有明确共享契约可复用。

### 2. 已对接现有 RAG 检索链路

`server/routes/rag.ts` 已通过 `runWebAigcSearch()` 统一处理 Web-AIGC 检索请求：

1. 校验请求
2. 用 `normalizeWebAigcSearchRequest()` 把请求映射到现有 `RetrievalOptions`
3. 调用 `deps.retriever.search()`
4. 用投影器格式化返回结果

`server/rag/web-aigc-search-adapter.ts` 已完成 Web-AIGC 到现有检索层的参数适配，包括：

- `projectId`
- `topK`
- `sourceTypes`
- `documentIds -> sourceIds`
- `agentId`
- `codeLanguage`
- `minScore`
- `mode`
- `expandContext`
- `contextWindowChunks`

`server/rag/retrieval/rag-retriever.ts` 则已经提供现有检索实现，支持：

- `semantic`
- `keyword`
- `hybrid`
- 多种过滤条件
- 上下文扩展

因此 `vector_query` 本质上已经落在现有 RAG runtime 上，而不是停留在 spec 层。

### 3. 结果里的元数据与分值已完整输出

结果输出并不只是“返回几段文本”，而是已经包含较完整的检索信息：

- 片段级：
  - `chunkId`
  - `documentId`
  - `sourceType`
  - `score`
  - `snippet`
  - `summary`
  - `highlight`
  - `positionHint`
  - `metadata`
- 文档级：
  - `documentId`
  - `sourceType`
  - `score`
  - `summary`
  - `highlights`
  - `fragments`

其中 `metadata` 来自 `server/rag/retrieval/rag-retriever.ts` 对 `metadataStore` 的组装，核心字段包括：

- `ingestedAt`
- `lastAccessedAt`
- `contentHash`
- 以及部分可选扩展字段，例如 `codeLanguage`、`functionSignature`

这已经满足“节点输出可继续被下游节点消费”的基本要求。

## 三、测试证据

### 1. Web-AIGC 路由适配测试已存在

`server/tests/rag-web-aigc-routes.test.ts` 已覆盖：

- `document-search` 结果按文档分组
- `fragment-search` 按 `documentIds` 过滤
- 缺失 `query` 时返回 `400`
- 缺失 `scope.projectId` 时返回 `400`
- 路由会把请求正确传给 `retriever.search()`

这说明 Web-AIGC 兼容层并非只写了类型，没有运行时验证。

### 2. 检索与结果结构相关底层测试也已存在

仓库里还存在以下检索相关测试：

- `server/tests/rag-retrieval.unit.test.ts`
- `server/tests/rag-retrieval.property.test.ts`

这些测试覆盖了：

- RRF 合并逻辑
- 上下文扩展逻辑
- `RetrievalResult` 所需字段完整性

说明 `vector_query` 复用的底层检索结果模型已有一定稳定性基础。

## 四、未完成部分

### 1. 查询监控事件尚未真正接入查询链路

虽然 `server/rag/observability/metrics.ts` 已定义：

- `recordRetrieval()`
- `rag_retrieval_total`
- `rag_retrieval_hit_rate`

并且 `server/routes/rag.ts` 暴露了 `/api/admin/rag/metrics`，
但仓库检索结果显示 `recordRetrieval()` 目前只有定义，没有被实际查询路径调用。

这意味着当前状态更像是：

- 有指标容器
- 有指标快照接口
- 但没有在 `document-search / fragment-search / search` 时真正记一次查询事件

所以还不能认为“查询监控事件已完成”。

### 2. Web-AIGC 节点级观测语义还没补齐

目前没有看到以下内容：

- `vector_query` 节点专用 observability 事件
- Web-AIGC 检索命中 / 未命中事件
- 文档搜索与片段搜索的专项监控测试
- 查询参数摘要、命中数量、耗时等字段的持久化审计

这一块如果后续要进主线收口，建议补在现有 RAG metrics 或 Web-AIGC audit/observability 体系上，而不是另起一套。

## 五、任务项结论

### 1. 定义向量查询输入输出

- 可以勾选
- 依据：
  - 共享请求/响应类型已定义
  - Web-AIGC 路由常量已定义
  - 基础校验与结果投影已存在

补充说明：

- 当前契约是“文本查询 + 检索选项 + 范围过滤”，不是“外部直接传原始向量”。如果 spec 后续要求原始向量入参，需要再扩展。

### 2. 对接现有 RAG 检索接口

- 可以勾选
- 依据：
  - 路由已实际调用现有 `retriever.search()`
  - adapter 已完成参数映射
  - 测试已验证调用发生

### 3. 输出元数据与分值

- 可以勾选
- 依据：
  - 返回结果里已有 `score`
  - 片段级结果里已有 `metadata`
  - 文档级聚合也保留了分值与片段信息

### 4. 写入查询监控事件

- 目前不建议勾选
- 依据：
  - 指标类存在
  - 但没有接入真实查询调用点
  - 缺少 Web-AIGC 检索监控专项测试

## 六、还差什么

要把这条 spec 完整收口，当前最小缺口主要有三类：

1. 把查询监控真正接到链路里
   - 在 `document-search / fragment-search / search` 成功或失败时写检索指标
   - 至少记录耗时、是否命中、结果数量

2. 明确 `vector_query` 的 Web-AIGC 观测语义
   - 复用现有 RAG metrics 或 Web-AIGC 审计体系
   - 不建议额外再造一套平行事件模型

3. 补专项测试
   - 文档搜索调用后指标递增
   - 片段搜索调用后指标递增
   - 命中率 / 耗时统计符合预期

## 七、结论

`web-aigc-node-vector_query` 当前已经具备“可用的查询薄切片”：

- 接口有了
- 检索链路打通了
- 返回结构能被消费了

但还不能说这条 spec 已经完全闭环，因为监控事件仍停留在“有指标类、未接调用点”的阶段。

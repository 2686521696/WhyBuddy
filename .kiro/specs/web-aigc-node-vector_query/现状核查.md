## 任务逐项判定

### 1. 定义向量查询输入输出

判定：已完成

依据：

- `shared/rag/web-aigc-search.ts` 已定义：
  - `WebAigcSearchRequest`
  - `WebAigcSearchScope`
  - `WebAigcSearchOptions`
  - `WebAigcDocumentSearchResponse`
  - `WebAigcFragmentSearchResponse`
  - `WebAigcSearchErrorResponse`
- 同文件还定义了 Web-AIGC 检索兼容接口常量：
  - `POST /api/rag/web-aigc/document-search`
  - `POST /api/rag/web-aigc/fragment-search`
- `server/rag/web-aigc-search-adapter.ts` 已实现请求归一化、结果投影与基础校验。
- `server/routes/rag.ts` 当前已对 `query`、`scope.projectId` 做请求校验并返回统一错误结构。

边界说明：

- 当前输入契约是“文本 `query` + 范围过滤 + 检索选项”。
- 当前实现不支持外部直接传入原始向量数组。
- 如果后续 spec 把“向量查询”严格定义为“可直接传入 embedding 向量”，则仍需扩展字段与校验。

### 2. 对接现有 RAG 检索接口

判定：已完成

依据：

- `server/routes/rag.ts` 中：
  - `/api/rag/web-aigc/document-search`
  - `/api/rag/web-aigc/fragment-search`
  都通过 `runWebAigcSearch()` 调用现有 `deps.retriever.search()`。
- `server/rag/web-aigc-search-adapter.ts` 已把 Web-AIGC 请求映射为现有 `RetrievalOptions`，包括：
  - `projectId`
  - `topK`
  - `sourceTypes`
  - `documentIds -> sourceIds`
  - `agentId`
  - `codeLanguage`
  - `minScore`
  - `mode`
  - `expandContext`
  - `contextWindowChunks`
- `server/rag/retrieval/rag-retriever.ts` 当前支持：
  - `semantic`
  - `keyword`
  - `hybrid`
  - 基于 `sourceType / sourceId / agentId / codeLanguage / timeRange` 的过滤
- `server/tests/rag-web-aigc-routes.test.ts` 已验证路由会把请求正确投递给 `retriever.search()`。

### 3. 输出元数据与分值

判定：已完成

依据：

- `shared/rag/web-aigc-search.ts` 中：
  - `WebAigcFragmentSearchHit` 已包含 `score`、`metadata`、`highlight`、`positionHint`
  - `WebAigcDocumentSearchHit` 已包含文档级 `score`，并内嵌片段结果
- `server/rag/retrieval/rag-retriever.ts` 会从 `metadataStore` 组装 `ChunkMetadata`，再回填到检索结果。
- `server/rag/web-aigc-search-adapter.ts` 会把检索结果投影为：
  - 片段级 `summary / highlight / positionHint / metadata / score`
  - 文档级 `score / highlights / fragments`
- `server/tests/rag-web-aigc-routes.test.ts` 已覆盖：
  - 文档聚合
  - 片段过滤
  - 摘要生成
  - 高亮投影

### 4. 写入查询监控事件

判定：已完成

依据：

- `server/rag/observability/metrics.ts` 虽然定义了：
  - `recordRetrieval()`
  - `rag_retrieval_total`
  - `rag_retrieval_hit_rate`
- `server/routes/rag.ts` 现在已经在以下真实查询路径中调用 `deps.metrics.recordRetrieval(latencyMs, hasResults)`：
  - `/api/rag/search`
  - `/api/rag/web-aigc/document-search`
  - `/api/rag/web-aigc/fragment-search`
- 查询成功时会按结果数量写入：
  - latency
  - 是否命中结果
- `server/tests/rag-web-aigc-routes.test.ts` 已新增断言并验证：
  - `document-search` 调用后会写 metrics
  - `fragment-search` 调用后会写 metrics
  - 通用 `/api/rag/search` 调用后也会写 metrics
  - 参数校验失败时不会误写 metrics
- `server/tests/rag-observability.property.test.ts` 继续证明指标对象本身的累计与快照逻辑成立

因此，这一项现在可以勾选。

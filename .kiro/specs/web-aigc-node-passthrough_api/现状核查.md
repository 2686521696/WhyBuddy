# 透传接口节点现状核查

## 结论

基于当前 main 工作区代码，`web-aigc-node-passthrough_api` 已经具备最小可运行闭环，可以按 spec 保守认定为已落地。

当前主仓不再只是“接近 `internal_api` 的思路”，而是已经形成了独立的 `passthrough_api` 执行器、`auto-agent` 目标种类接入、白名单与超时控制、响应脱敏与审计记录，以及失败/超时测试覆盖。因此 `tasks.md` 中 4 个任务项现在都可以勾选。

## 已落地能力

依据：

- `server/tool/api/passthrough-api-adapter.ts`
  - 已定义 `PassthroughApiExecutionRequest`

- `server/tool/api/auto-agent-adapter.ts`
  - `AutoAgentTargetKind` 已包含 `passthrough_api`
  - 已新增 `executePassthroughApi(...)` 分支
- `server/routes/a2a.ts`

- `server/tests/auto-agent-adapter.test.ts`
- `server/tests/auto-agent-routes.test.ts`
  - 已有 `passthrough_api` 的执行入口测试

结论：`passthrough_api` 不再是“未定义目标种类”，而是已经具备独立请求/响应契约和统一入口路由。

### 2. 已落地白名单与超时策略

依据：

- `server/tool/api/passthrough-api-adapter.ts`
  - 已要求 `metadata.whitelist`

  - 已支持 `metadata.timeoutMs`
  - 已对超时中断映射为显式错误
- `server/tests/passthrough-api-adapter.test.ts`
  - 已验证白名单拒绝

结论：`passthrough_api` 已具备节点自身的白名单与超时治理，而不是仅依赖仓库里散落的通用基础件。

### 3. 已落地结果脱敏与审计

依据：

- `server/tool/api/passthrough-api-adapter.ts`
  - 已复用 `maskSensitiveData`

- `server/tests/passthrough-api-adapter.test.ts`
  - 已验证邮箱与 token 脱敏
  - 已验证 `allowed / denied / error` 审计分支

结论：当前 `passthrough_api` 已不只是“可复用脱敏基础件存在”，而是节点自身已经接上结果脱敏与审计闭环。

### 4. 已验证失败与超时场景

依据：

- `server/tests/passthrough-api-adapter.test.ts`
  - 已验证白名单拒绝

- `server/tests/auto-agent-routes.test.ts`

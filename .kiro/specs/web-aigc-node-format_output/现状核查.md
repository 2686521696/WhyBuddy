# 现状核查：格式输出节点

## 核查范围

- 节点适配器：`server/routes/node-adapters/format-output-node-adapter.ts`
- 路由入口：`server/routes/format-output.ts`
- 测试：
  - `server/tests/format-output-node-adapter.test.ts`
  - `server/tests/format-output-routes.test.ts`

## 当前结论

`web-aigc-node-format_output` 已完成最小可运行闭环，当前任务清单可按 `4 / 4` 认定完成。

当前已具备以下能力：

1. 支持 `text / json / table / template` 四类输出格式。
2. 对外提供独立节点执行入口：`POST /api/format-output/nodes/execute`。
3. 当模板渲染失败、表格输入不兼容等情况发生时，会自动回退为 `raw/data` 的文本输出，并返回错误说明。
4. 输出结构统一包含 `content`、`structured`、`fallbackUsed`、`metadata.downstreamConsumers` 等字段，可直接作为 `end`、`file_generation` 这类下游节点的消费输入。

本轮收口聚焦于“节点级最小闭环”，即独立路由、节点适配器、失败回退与下游消费契约；不把 `server/index.ts` 里的批量挂载热区混入本次提交。

## 事实依据

### 1. 输出格式集合已定义

节点适配器中已经明确支持以下格式：

- `text`
- `json`
- `table`
- `template`

并对非法格式返回明确错误，避免静默失败。

### 2. 格式转换策略已落地

当前转换行为如下：

- `text`
  - 将 `data` 或 `raw` 规范化为可展示文本。
- `json`
  - 保留结构化对象，同时输出可直接落盘或展示的 JSON 字符串。
- `table`
  - 支持对象数组转换为 Markdown 表格，并返回列、行等结构化信息。
- `template`
  - 支持 `{{key}}` 与 `{{nested.path}}` 形式变量替换，并保留模板变量清单。

### 3. 失败回退逻辑已落地

当前以下场景会回退为原始文本输出：

- 模板缺少必需变量
- 模板正文缺失
- 表格输入不是数组或不满足表格渲染前提
- 其它格式化过程抛错

回退后仍返回 `status: completed`，同时标记：

- `fallbackUsed: true`
- `format: text`
- `metadata.fallbackFrom`
- `error`

这保证了下游节点即使遇到格式化异常，也仍能消费一个稳定字符串结果。

### 4. 与结束节点、文件生成节点的联动已被测试证明

当前没有直接改动 `end` 或 `file_generation` 主线代码，但已通过输出契约证明其可被下游消费：

- 所有格式最终都产出稳定的 `content: string`
- 同时保留 `structured` 结构化数据，便于下游按 JSON 或表格继续处理
- `metadata.downstreamConsumers` 已固定标注 `end`、`file_generation`

相关测试覆盖：

- 纯文本输出
- JSON 输出
- Markdown 表格输出
- 模板输出
- 模板失败回退
- 表格失败回退
- 路由层非法 `nodeType`
- 路由层非法 `format`
- 路由层成功返回可供下游消费的表格结果
- 路由层模板失败后的原始文本回退

## 已知边界

本次完成的是“最小可运行闭环”，不是 runtime 主干深度集成版。

当前未做的事情：

- 未改 `workflow-runtime-engine`
- 未把 `format_output` 接入更高层 workflow 图执行分发
- 未把 `format_output` 的主应用挂载从 `server/index.ts` 热区中单独拆出并纳入本次提交
- 未扩展富模板能力，如条件分支、循环渲染、复杂 helper

因此，这条 lane 的完成定义应理解为：

- 节点输入输出契约已成立
- 独立执行入口已成立
- 失败回退已成立
- 下游可消费性已被测试证明

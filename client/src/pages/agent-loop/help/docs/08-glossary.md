# 术语表

SlideRule 界面和文档里出现的专业词，都在这里说白话。

## 推演

不是"直接生成"，而是像一支产品团队那样**多轮论证**：理解意图 → 找证据 →
问清楚 → 挑毛病 → 收敛结论。你在左栏看到的直播就是推演过程本身。

## 五系统模型

一个企业应用被拆成五个互相咬合的系统，外加一层总装：

| 系统 | 管什么 | 举例 |
| --- | --- | --- |
| 数据模型 DataModel | 业务对象和字段（唯一事实源） | 采购单、金额、状态 |
| 角色权限 RBAC | 谁能看什么、做什么 | 经理能审批，员工只能提交 |
| 工作流 Workflow | 事情按什么顺序流转 | 提交 → 审批 → 确认 → 完成 |
| 页面 Page | 用户在哪操作 | 工作台、看板、日历、仪表盘 |
| AI 能力 AIGC | 哪些环节由 AI 帮忙 | 识别发票、评估风险 |
| 总装 AppBundle | 把五系统装配成应用 | 页面绑流程、角色配菜单 |

## 闭环 / closed 6/6

六个系统的证据全部齐备、总装通过——徽标 `closed 6/6` 亮起，应用才算
"端出来了"。少一段就是 blocked（被拦）。

## 确定性门

机械的、不靠 AI 自觉的检查关卡。**结构门**查模型六段齐全、跨系统引用
全部真实可解析；**覆盖门**查六系统证据齐备。AI 说"我做完了"没用，
过了门才算数——这是本产品的北极星。

## 证据 / 溯源（provenance）

推演每一步的产物都带来源标记：是 LLM 生成的、全网检索来的、还是沙盒里
真实执行出来的。右侧「发布证据看板」能看到每段证据的指纹（digest）。

## 数字孪生

推演产出的不是原型图，而是带真实数据模型、权限、流程的**可操作应用**
——你可以切角色、录数据、走流程，像用真系统一样验证这门生意能不能转。

## 智能编排（agentic pick）

每轮推演"下一步做什么"由 AI 看全局仪表盘自主提案，但套着三道保险：
停不停仍归规则管（AI 无权无限续命）、只能从固定能力菜单点菜、AI 失灵
自动回落规则版。评测 4:0 胜出后已默认开启。

## 增量精修

闭环后再发补充指令（如"加一个志愿者角色"），引擎在现有模型基础上做
**最小增量修改**——无关字段逐字节保留，绝不推倒重来。

## 版本前进/回退

每次模型真实变化都自动存档为一个版本，工具栏 ◀ ▶ 可回到任意版本再前进。
存档在服务端，刷新、换设备都不丢。

## 缺口修复轮

被闸拦截后点「补齐缺口」，只重跑覆盖门标红的部分——哪里缺补哪里，
不是整轮重来。

## 断线重生

推演跑在服务端。中途关页面、断网、换设备，回来自动接回直播，从断点
继续看，结果不丢。

## 内置演示域

采购审批、请假审批、服务工单、员工入职四个域走内置模板秒出结果（不调
AI），用于快速体验。它们的五系统模型是预先生成并过门冻结的。

## 附件提取

上传的文件如何变成推演上下文：文本类直接读取注入；**图片**由视觉 AI
识别内容；**PDF** 送进一次性云沙盒解析文本（本机零执行）。解析失败会
如实说明，只随消息带文件名，绝不假装解析成功。

## 沙盒（E2B）

需要真实执行代码的环节（如验证取证、解析 PDF）都在一次性云沙盒里跑，
跑完即销毁——你的机器上永远不执行任何生成代码。

## 技能库 / MCP

可安装的扩展能力包。勾选后随推演注入，让引擎按技能说明书行事；MCP 是
接入外部工具（如全网检索）的标准协议。

## 交付物

推演的文字成果：可行性报告、任务清单、需求追溯矩阵、交接包等，右上角
「交付物」面板查看并一键导出 Markdown。

## 推演通道

驱动推演的大模型接入配置（任意 OpenAI 兼容供应商）。在「设置 → 推演
通道」可运行时更换，密钥只存本机服务端、界面永远只显示掩码。

# 引擎内幕（进阶）

想知道推演引擎内部怎么运转的，看这一节。

## 能力（capability）与选材（pick）

推演的最小执行单元叫**能力**——澄清需求、检索证据、分析风险、红队质疑、
综合结论、写报告……每轮开始引擎要**选材**：这轮派哪几个能力上场。选材有
两套：**规则版**（关键词路由 + 缺啥补啥，确定性）和**智能编排**（LLM 看
全局提案，见上文）。无论谁选的，每个决策都进决策台账。

## 决策台账（decision ledger）

每轮"看到了哪些候选、选了哪些、为什么"的完整记录，LLM 提案的决策标注
`source=llm`——可审计、可挑战，AI 的编排不是黑箱。

## 注入 Prompt

一个能力执行时，它的 LLM 提示词里实际装了什么：你的目标原文 + 本轮
指令 + **装箱后的上游证据**（见下条）+ 勾选的技能说明书 + 附件提取内容。
所见即所喂——推演页左栏能看到每步吃了什么。

## 证据装箱（证据上下文管道）

轮内推理能力（综合、报告、反证…）的原料不是凭空来的，而是把**上游健康
产物**装进它的 prompt。装箱有三道规矩：

- **准入 = 信任门**：只收过了门（gated_pass/audited）且未失效的产物
  ——只喂过门的，不喂垃圾
- **优先级**：按能力口味排序（综合优先吃风险和证据；报告优先吃已综合的
  结论），超预算时整件丢弃低优先级，不腰斩
- **诚实省略**：预算装不下的，留一行「另有 N 件…未注入」——省略了就说

没有这条管道时，"综合各方结论"综合的其实是对话题的独立再推演（实测
挖出的缺陷，2026-07-16 修复）。

## 轮内屏障

同一轮里，普通能力并行跑，但**吃上游产物的能力**（综合/报告）必须等
前面的能力产物真正落库后才执行——它们在执行序列里作为"屏障"串行卡位。
没有屏障时报告可能引用一个还没写完的结论。

## 结构化回喂

LLM 生成的五系统模型没过结构门时，不做盲目重试——把门的**具体裁决**
（哪个引用悬空、哪段缺失）喂回给 LLM 重新生成，错哪改哪。

## 确定性修复

门禁之前先跑一遍零 LLM 的机械修复：模型里不变式的悬空引用做近邻改写
（唯一命中才改），修不好的整条剔除并留痕（应用总装屏如实展示剔除记录）。

## 闭环指纹

证据看板底部那行 `closureHash / digest / versionPins`：闭环的内容指纹。
六段证据的信任字段哈希出唯一指纹——同一指纹 = 同一闭环，任何一段证据
变化指纹必变。模型段是纯载荷，不进指纹。

## stale（失效）与挑战

对某轮结论点「挑战」，相关产物被标记 stale（失效），依赖它的下游产物
级联失效——下一轮推演必须重新生产，不能拿旧结论蒙混。

## run 与事件日志

每次推演是服务端的一个 **run**：全部过程事件按序号写进事件日志。断线
重连时客户端说"我看到第 N 条了"，服务端从 N+1 续播——这就是断线重生的
底层机制。孤儿 run 有看门狗兜底回收。

## A/B 评测

引擎的重大默认值切换不靠感觉，靠对照实验：同一批话题分别用新旧两种
配置各跑一遍（如智能编排 on/off、证据装箱 on/off），再由 **LLM 裁判**
盲评产物质量。智能编排就是十话题对比 + 内容质量评测 4:0 胜出后才转正的。
每个机制都留有停用开关，作为对照臂随时可复验。
